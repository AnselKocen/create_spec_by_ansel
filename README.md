# Spec Forge 使用说明

Spec Forge 是一个在 Codex 里使用的 spec 生成流程。它生成的是可填充的玩法 / 产品结构：后续管线会把 `spec.md` 和 `skills/内容文件` 结合起来，填入具体主题、角色、章节、题目、条目、插图和文本。

你不需要记命令，不需要配置 API key，也不需要自己选择一堆参数。只要在 Codex 客户端里打开这个项目，然后告诉 Codex 你要做什么 spec。

---

## 唯一使用方法

### 1. 在 Codex 里打开这个项目

项目路径：

```text
/Users/anselkocen/Desktop/spec合集
```

### 2. 直接对 Codex 说你想做什么

你不需要先准备随附内容文件。最推荐的开局是：

```text
按 specforge 流程，帮我做一个 spec。先问我想做什么方向。
```

也可以稍微给一点方向：

```text
按 specforge 流程，我想做一个互动学习游戏 spec。先问我玩法和方向。
```

或者：

```text
按 specforge 流程，我想做一个多结局文字冒险 spec。我们先聊玩法结构，不要直接写最终 spec。
```

Codex 会先读取这个项目里的规则和参考 spec，然后直接开始第一轮问题。它不需要先复述规则或列执行清单，而是通过几轮自然语言提问，把方向、玩法结构、生图、准确性和视觉风格问清楚。最终 spec 的核心结构是固定的，访谈主要决定内容如何被 `skills` 填充、玩法如何组织、视觉如何约束。

开局时 Codex 不会一次性丢出所有问题，而是分几轮问。第一轮通常先确认：

```text
1. 你想要什么玩法 / 产品结构？
2. 你希望最终用户体验一个完成品，还是使用一个生成工具？
3. 后续会随 spec 一起进入管线的 skills/内容文件，大概是什么内容类型？
4. 目标用户是谁？
```

如果你只说“阅读 specforge”，Codex 读完后也应该直接问这一轮问题。

### 3. 如果当前项目里已经有 skills/内容文件，可以告诉 Codex 先读取

```text
按 specforge 流程，读取 ./skills，做一个分支叙事选择游戏 spec。
```

或者：

```text
按 specforge 流程，读取 ./skills，做一个图鉴收集 spec。
```

也可以更口语一点：

```text
我想用这些内容文件做一个人格测试类 spec，按 specforge 流程来。
```

所有最终 spec 都默认会和完整 `skills/内容文件` 一起进入后续管线。当前对话里即使还没有实际文件，也不需要问“有没有”；Codex 应该先通过访谈确认这些文件会提供什么类型的内容、会填入哪些槽位。

但最终生成的 `spec.md` 必须说明：下一步实现时，`spec.md` 会和完整内容文件一起提供，通常是 `skills/` 目录；实现模型必须读取这些文件的完整内容，并把它们融入产品。

## 两种产品方向

这不是两种使用方法，只是最终 spec 的两种产品方向。你仍然只需要在 Codex 里自然语言开局。

| 模式 | 什么时候选 | 最终产品是什么 |
|------|------------|----------------|
| 可填充成品框架 | 默认选择。你希望后续管线用 `spec + skills/内容文件` 做出一个完成品 | 用户打开后直接玩、读、测试、收集或学习 |
| 生成工具框架 | 当你明确想做“生成器 / 工具 / 创作器 / 改编器”时选择 | 用户打开后输入内容、选择参数、生成、预览、导出 |

两种模式都会保留 UI 设计、粒子氛围、SVG/CSS 技巧、生图约束、内容文件契约、数据 Schema 和验收标准。如果产品要求高精度 SVG / Canvas / 互动模拟，最终 spec 还必须包含领域专项绘制指南，不能只写“高精度”“酷炫”“分层”。

区别只在于：可填充成品框架的“读取内容文件、提取内容、填充数据”发生在后续管线 / 构建阶段；如果需要真实图片，默认使用 `first-run-cache`：首次打开先查 IndexedDB，cache hit 读取 Blob、创建本次会话 object URL、加载并解码，cache miss 才显示资产准备页、调用真实生图、把图片数据转成 Blob 写入 IndexedDB，再创建 object URL、加载并解码；如果生图工具返回 URL，必须先确认该 URL 在当前页面可读取并可转成真实图片 Blob，再进入缓存和 ready 流程；如果生图、URL 转 Blob、加载或解码失败，允许用户跳过资产准备页进入显式无图片体验，失败资产保持 failed / ready=false 并提供重试。之后同一浏览器 / 同一域名刷新或再打开只读 IndexedDB Blob 缓存，不重新生图。生成工具框架则把输入、生成、预览、导出作为最终用户可操作的核心功能；用户生成的结果图也应写入 IndexedDB / 历史记录，刷新后已有结果不得无故重新生成。

如果你不确定，就选可填充成品框架。`fixed-content` / `generator-tool` 只作为 Spec Forge 内部模式或玩法方案信息；最终 `spec.md` 应用自然语言描述产品形态，不直接暴露这些内部标签。

## Codex 会怎么做

Codex 不会一上来直接写完整 spec。

它会分几轮问清楚关键问题。一般是：

第一轮：方向和产品形态

1. 你想要什么玩法 / 产品结构？比如分支叙事、图鉴收集、互动模拟、问答测试、绘本阅读、解谜、卡牌组合等。
2. 你希望最终用户体验一个完成品，还是使用一个生成工具？
3. 后续随 spec 一起提供的 `skills/内容文件` 大概是什么内容类型？
4. 目标用户是谁？

第二轮：玩法结构和内容槽位

1. 玩家或用户主要怎么玩 / 怎么用？
2. 哪些内容由 `skills/内容文件` 填入？例如主题、角色、章节、题目、图鉴条目、结局、插图 prompt。
3. 是线性推进、自由探索、分支路线、循环玩法，还是模拟实验？
4. 内容规模大概是多少？

第三轮：生图和视觉

1. 要不要真实 AI 生图？如果要，哪些部分需要图：标题背景、章节插图、角色、图鉴条目、卡牌、结局 CG 等。
2. 默认图片使用 `first-run-cache`：首次打开先查 IndexedDB，cache hit 读取 Blob、创建 object URL、加载并解码，缺图才生成、转 Blob 写入 IndexedDB、再加载解码，之后同一浏览器 / 同一域名刷新或再打开不重新生图；如果生图返回 URL，需要确认它可读取并可转成图片 Blob；如果生图失败，是否允许用户跳过资产准备页进入无图片体验；有没有少数图片必须实现时先生成并打包，或需要混合模式？
3. UI / 画风 / 氛围有什么偏好？如果没有，Codex 会按媒介、情绪、材质、光影、构图等维度推导。

第四轮：准确性和硬约束

1. 内容需要严谨校验，还是允许幻想发挥？
2. 哪些内容必须严格来自 `skills/内容文件`？
3. 还有没有必须保留的硬约束？

然后 Codex 会先生成：

```text
generated-specs/项目名/spec.gameplay.md
```

这是单独的玩法结构方案。你需要先确认核心循环、内容槽位和映射方式。

确认后，Codex 才会继续生成：

```text
generated-specs/项目名/spec.md
```

这是最终 spec。

如果你没有指定名字，Codex 会根据产品标题或主题自动取一个短名字。已有同名目录时，Codex 应该先问你是否覆盖，或者另取一个版本名。

最终 `spec.md` 会自动包含“涉及的 SKILL / 内容文件清单”。这部分会告诉下一步实现模型：完整内容文件会与本 spec 一起提供，通常在 `skills/` 目录；实现时必须读取全文，并从中提取本产品需要的内容。

## 保存规则

生成的 spec 不应该直接散落在项目根目录。

默认保存到：

```text
generated-specs/项目名/
```

目录里包含：

```text
spec.gameplay.md
spec.md
```

例如你要做“深空物理实验室”，输出应类似：

```text
generated-specs/深空物理实验室/spec.gameplay.md
generated-specs/深空物理实验室/spec.md
```

如果你还没想好名字，可以直接说“你帮我取”。Codex 会根据玩法和主题取一个短而明确的目录名。

## 为什么要先确认玩法

因为 spec 的质量主要取决于玩法是否立住。

如果一开始就写完整 spec，模型很容易一边写 UI、一边改玩法，最后文档看起来很满，但核心循环不稳。

所以 Spec Forge 固定流程是：

```text
读取项目规则 → 分轮访谈 → 确认玩法结构和产品方向 → 如有随附内容文件则读取 → 确认保存名称 → 生成玩法结构方案 → 用户确认 → 生成最终 spec
```

## 最终 spec 会保留什么

最终 spec 会复用这个项目里已经验证过的关键约束：

- 固定章节骨架
- 设计粒度：可填充玩法框架
- 内部产品形态区分：可填充成品框架 / 生成工具框架，内部标签只出现在玩法方案或规划中
- 内容提取和玩法映射方法
- 随附内容文件 / SKILL 契约
- 生图限制
- UI 美化要求
- SVG / CSS 技巧
- 高精度 SVG / CSS 可视化原语、质感配方、交互态和常见画错点
- 粒子和氛围设计
- 动效标准
- 数据 Schema 写法
- 内容准确性模式
- 验收标准

但参考 spec 只是“固定骨架的模式库”，不是整篇照抄模板。

不同 spec 应该保留同一套核心结构，但生成不同玩法结构、内容槽位、视觉方法、数据字段和内容文件映射。参考 spec 提供结构能力，可以组合使用。

固定骨架见：

```text
specforge/modules/fixed-spec-skeleton.md
```

## 生图规则

最终 spec 固定会有“生图策略与资产契约”章节。

如果这个产品不需要真实位图资产，章节里会写清不需要的范围。

如果这个产品需要真实生图，最终 spec 会写入强约束：

- CSS 渐变不是生图
- SVG path 不是生图
- Canvas 绘制不是生图
- emoji / Unicode 不是生图
- 文字占位不是生图
- 需要说明哪些部分生图
- 需要声明生图时机；默认是 `first-run-cache`，只有明确需要时才使用构建期生成 / 混合
- 首次打开必须先查 IndexedDB；cache hit 只读取 Blob、创建本次会话 object URL、加载并解码，不显示完整生成进度；cache miss 才显示“正在生成图片 / 正在准备视觉资产”的资产准备页
- 如果生图工具返回 URL，必须先确认该 URL 在当前页面可读取、可转成真实图片 Blob，并通过 MIME / load / decode 校验；失败的 URL 不得当作已生成或已缓存图片
- 生成后的图片 Blob 必须写入 IndexedDB；required 图片必须在进入正常图片体验前完成 `generated + cached/cache_hit + loaded + decoded + ready`，正常图片体验阶段不得逐页生成、逐页请求或逐页等待 required 图片
- 如果生图、URL 转 Blob、加载或解码失败，允许用户跳过资产准备页进入显式无图片体验；失败资产必须保留 failed / ready=false 状态，并提供重试入口
- localStorage 只能保存轻量进度和状态，不能保存图片、base64、大 data URL、Blob 字符串或 object URL
- 必须有统一画风基底
- 每张图必须写详细 prompt
- 必须有静态 `IMAGE_ASSET_MANIFEST` 和运行时 `IMAGE_ASSET_RUNTIME_STATE`
- 无法生图时不能假装完成图片体验，不能用占位图冒充生成结果；可以交付显式标记的无图片体验

这些限制是为了防止下一步模型偷懒。

## 准确性模式

Codex 会问你内容准确性要求。

一般可以这样选：

| 模式 | 适合内容 |
|------|----------|
| 严谨 | 物理、化学、历史、真实人物、医学、法律 |
| 半严谨 | 科普、轻学习、历史改编 |
| 幻想 | 恐怖、奇幻、人格测试、抽象玩法 |

严谨模式会要求公式、方程式、时间线、人物关系、实验现象等都必须自检。

幻想模式允许发挥，但要保持主题一致、体验完整。
