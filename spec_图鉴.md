### ⚠️ 核心设计理念

> **这是一本活的图鉴，不是一套题库。**
>
> 玩家的目标是收集一本完整的图鉴——每一张卡片都是「答对知识 → 获得精美插图」的奖励。知识是钥匙，图鉴是宝藏。去掉图鉴只剩答题，这个产品就没有存在价值。
>
> **本产品的核心难度在 UI 设计，不在游戏机制。** 图鉴卡片、画廊、分享卡片必须精致到让人想截图收藏。视觉风格由图鉴内容的领域决定——自然类要森系、生物类要博物志手绘感、人文类要复古文艺——风格必须统一且贯穿全局。

## 四、视觉风格规范

> **本节是整份 spec 中最重要的部分。** 图鉴游戏的价值一半在知识，一半在视觉。卡片不够美，就没有收集欲。

### 整体气质（强制：由主题决定，不可套用默认风格）

> **禁止使用默认的"牛皮纸/羊皮纸/古籍"风格。** 视觉风格必须根据 skills 内容的主题领域专门设计。

**视觉风格确定流程（强制，必须在设计 UI 之前完成）**：

1. **分析 skills 内容**：读取所有 skills 文件，判断图鉴主题属于什么领域（如海洋生物、城市建筑、中国美食、太空探索、民间传说……）
2. **选定视觉风格关键词**：根据领域选一个有辨识度的风格方向，这个关键词将贯穿 UI 设计、配色、卡片样式、生图提示词
3. **确定主色系**：根据主题选定一个主色系（如海洋→深蓝、植物→森绿、美食→暖橙、太空→深紫），整个 UI 围绕这个主色系展开
4. **设计 UI 气质**：所有 UI 元素（卡片、按钮、背景、字体风格）必须与主题和主色系统一

**示例**（仅供参考方向，不可直接照搬）：

| 主题 | 风格方向 | 主色系 | UI 气质 |
|------|---------|--------|---------|
| 海洋生物 | 深海探索感 | 深蓝+青色 | 气泡、波光、半透明面板 |
| 植物花卉 | 森系水彩 | 森绿+米白 | 叶脉纹理、水彩渗透、自然边缘 |
| 城市建筑 | 建筑蓝图感 | 深蓝灰+白线 | 网格、精确线条、工程图质感 |
| 中国美食 | 国潮插画 | 朱红+金色 | 印章、毛笔字体、中式纹样 |
| 太空天体 | 科幻深空 | 深紫+荧光蓝 | 发光边框、星云背景、全息感 |
| 民间传说 | 复古版画 | 赭石+墨色 | 版画纹理、粗糙纸质、古朴边框 |

> **关键要求**：看到 UI 就应该能猜到这本图鉴是关于什么的。如果把标题遮住，玩家也能从配色和质感感受到主题。

### 封面页设计（固定功能，必须实现）

> **游戏的第一个画面是封面页，不是输入用户名。** 封面页是玩家对这本图鉴的第一印象，必须有仪式感。

**封面页内容**：
- **AI 生成的封面背景图** — 根据图鉴主题和画风基底编写提示词，调用生图模型生成一张精美的封面图
- **游戏标题** — 主题化字体设计，浮在背景图上方
- **"点击任意位置进入"提示** — 小字，有呼吸闪烁动画

**封面图要求**：
- 封面图仅用于封面页，不用于其他界面
- 封面图作为最底层图层，标题和提示文字浮在其上
- 封面图整体氛围与图鉴主题一致，可以稍暗或加遮罩保证文字可读

**点击后的过渡动画**：
- 点击任意位置后，用主题化的过渡动画进入欢迎页（输入用户名）
- 过渡方式由开发者根据主题选择，例如：
  - 翻书效果 — 适合博物志、古典主题
  - 穿越/缩放效果 — 适合科幻、太空主题
  - 水墨晕开效果 — 适合中国风主题
  - 渐隐渐显 — 通用
  - 卷轴展开 — 适合历史、地图主题
- 过渡时长 1-2s，要有仪式感

### 背景系统

> 封面页之外的其他界面不使用全屏背景图。画廊、答题等界面使用主色系纯色/渐变背景，让图鉴卡片成为视觉焦点。

> 原则：背景是氛围的一部分，必须与图鉴整体风格一致。所有注意力焦点在前景的图鉴卡片和 UI 上。

### 标题与品牌设计

> **游戏标题不是普通的 `<h1>` 文字。** 标题是整个图鉴的品牌，贯穿画廊顶部、分享卡片、结局页，必须有主题化设计。

- 字体效果必须与图鉴主题匹配（用 CSS 实现：`text-shadow` 发光/阴影、`background-clip: text` 渐变、`-webkit-text-stroke` 描边、`letter-spacing` 调节等）
- 可用 SVG 装饰元素围绕标题（如藤蔓、波浪、齿轮、星星等，取决于主题）
- 画廊页顶部的标题应有入场动画（渐显、缩放等）
- 副标题/进度信息用更小更轻的字体，与主标题形成层次

### 粒子/氛围装饰效果

> **画廊页不能是"一个标题 + 一堆卡片"的白板。** 必须有与主题匹配的氛围装饰，让整个界面有"活"的感觉。

根据图鉴主题选择 1-2 种氛围效果：

| 效果 | 适用主题 | 实现要点 |
|------|---------|---------|
| 花瓣飘落 | 植物/春天/日系 | SVG 花瓣 + CSS animation，随机起点/速度/旋转 |
| 星星闪烁 | 太空/夜晚/魔法 | CSS opacity + scale 脉动，散布在背景上 |
| 气泡上浮 | 水下/海洋/饮品 | CSS animation 上浮 + 微弱左右摆动 |
| 萤火虫漂浮 | 森林/夏夜/童话 | CSS 光点 + JS 随机缓动路径 |
| 落叶旋转 | 秋天/自然/怀旧 | SVG 叶子 + CSS transform rotate |
| 光斑漂移 | 梦幻/水面/森林 | CSS radial-gradient 大光斑，极慢漂移 |
| 雪花飘落 | 冬季/冰雪/北欧 | CSS animation 竖向 + 微弱横向漂移 |
| 尘埃漂浮 | 古老/图书馆/考古 | 极小 CSS 圆点，低透明度，缓慢随机漂浮 |
| 几何碎片 | 抽象/极简/现代 | CSS 三角形/圆形，缓慢旋转漂浮 |
| 音符飘浮 | 音乐/乐器 | SVG 音符 + CSS animation 上浮 |
| 烟雾缭绕 | 美食/茶道/香料 | CSS radial-gradient 模糊色块，极慢漂移 |
| 像素粒子 | 复古/游戏/数码 | 小方块，8bit 风格步进式运动 |

实现约束：
- 粒子数量 10-20 个，不影响性能
- 粒子是氛围层，不应遮挡卡片内容
- 尊重 `prefers-reduced-motion`

### SVG / CSS UI 设计技巧

> 鼓励使用 SVG 和 CSS 来设计主题化的 UI 元素，不要只用默认样式。

**SVG 适合做的事**：
- 主题化的卡片边框装饰（藤蔓、波浪、齿轮、像素边框等，用 `<path>` 绘制）
- 类别图标（用 SVG 而非 emoji，风格统一）
- 进度条/收集条（SVG 弧形进度、分段进度条）
- 解锁动画中的装饰粒子形状（星星、花瓣等用 SVG 而非圆点）
- 空状态/未解锁的占位图案（主题化的剪影或图案）

**CSS 设计技巧**：
- 卡片用 `clip-path` 裁剪非矩形外形（如圆角六边形、书页形、邮票锯齿边等，取决于主题）
- 卡片 hover 用 `transform: translateY(-4px)` + `box-shadow` 加深，营造浮起感
- 用 `backdrop-filter: blur()` 做半透明信息面板
- 用 `background-clip: text` + 渐变做主题化的标签/标题文字
- 未解锁卡片用 CSS `filter: brightness(0)` 做剪影 + `filter: blur(2px)` 做模糊
- 解锁瞬间用 CSS `@keyframes` 做翻转（`rotateY(180deg)`）+ 光效扩散（`box-shadow` 动画）

### 整体 UI 设计要求

> **UI 和主题是一体的。** 卡片形状、网格布局、装饰元素、字体气质都应该从主题中生长出来，不只是换个颜色。所有界面禁止使用浏览器默认样式，必须经过主题化设计。

**UI 形态必须随主题变化的维度**：

| 维度 | 不能只换色 | 应该做到 |
|------|----------|---------|
| 卡片形状 | 全是圆角矩形 | 根据主题选择形状（CSS `clip-path`）：贝壳、叶片、六边形、书页、邮票锯齿等 |
| 网格布局 | 全是等距网格 | 可以是瀑布流、蜂巢排列、书架式、散落式 |
| 分隔方式 | 全是直线 | 可以是波浪、藤蔓、虚线、裂纹（SVG 绘制） |
| 字体气质 | 全是无衬线 | 古典主题用衬线、手写主题用手写风、科技用等宽 |
| 装饰元素 | 没有 | 界面角落/边缘有主题装饰（花纹、图标、纹理） |
| 未解锁态 | 全是灰色方块 | 根据主题设计：迷雾笼罩、信封未拆、化石待挖掘、像素马赛克等 |

### UI 元素风格

**图鉴卡片（核心元素）**：
- 卡片是玩家收集的核心奖励，必须精致
- 插图区域占比 ≥ 60%，插图是主体
- 卡片形状必须与主题风格一致，不要所有主题都用圆角矩形
- 解锁动画：翻转/光效/粒子，要有仪式感
- 卡片底色和边框配色必须使用主色系

**卡片固定信息字段**（每张卡片必须展示）：
- **条目名称**（中文名）— 主标题，字号最大
- **专有名词**（学名/希腊名/英文名/术语等，取决于主题）— 副标题，斜体或小字
- **条目简介**（1-2 句核心描述）— 正文字号
- **类别标签** — 小标签样式，用主色系辅助色

**未解锁卡片**：
- 必须有主题化的未解锁设计，不只是灰色方块
- 保持神秘感，让玩家想要解锁

**画廊网格**：
- 桌面端 3-4 列，平板 2-3 列，手机 2 列
- 网格布局形式可根据主题调整（不必拘泥于等距网格）
- 未解锁卡片与已解锁有明显区分
- 进度展示（n/N）要醒目

**按钮**：
- 按钮配色使用主色系的强调色
- hover/active 状态必须有明确的视觉反馈
- 按钮的形状和质感与整体 UI 气质一致

**页面过渡**：
- 画廊 ↔ 答题 ↔ 卡片展示之间的切换必须有过渡动画
- 时长 300-800ms，不能瞬切

### 配色方案

> 配色由开发者根据主题和主色系自行决定，遵循以下原则：

- **60-30-10 法则**：60% 主色（背景/表面）、30% 辅助色（文字/边框）、10% 强调色（按钮/高亮）
- **主色系必须来自主题**
- 图鉴插图是画面中最丰富多彩的部分，UI 配色应衬托而非争抢
- 成功色和错误色可以通用（绿/红），但也应与主色系协调

### 必须实现的视觉反馈

| 用户操作 | 视觉反馈要求 |
|---------|------------|
| 点击未解锁条目 | 进入该条目的答题界面，过渡动画 |
| 答对解锁 | 卡片翻转/展开动画 + 插图显现 + 庆祝光效 + 「图鉴 GET!」提示 |
| 答错 | 温和提示 + 可重试，不惩罚 |
| 浏览画廊 | 卡片 hover 微浮起 + 阴影加深，已解锁的显示缩略图 |

| 屏幕切换 | 淡入淡出过渡，不能瞬切 |
| 悬停可交互元素 | 发光/放大/颜色变化，必须有明确反馈 |

### 动效标准

- 所有 UI 过渡时长 200-800ms
- 使用 ease 或 cubic-bezier 缓动曲线，禁止线性运动
- 背景动画周期 ≥ 15s，极缓慢
- 卡片解锁动画是核心奖励感来源，时长 1-2s，要有仪式感
- 尊重用户的 `prefers-reduced-motion` 系统偏好
- 动效必须有目的，禁止无意义的闪烁

## 九、交互设计

### 交互模式选择

**本项目选择**：自由探索 + 多样答题

**选择原因**：图鉴收集本身是非线性的，玩家应有自由选择的权利。答题形式根据知识内容适配，避免单一选择题的疲劳感。

### 操作映射

| 操作 | 实现方案 | 说明 |
|------|---------|------|
| 输入用户名 | 文本输入框 | 首次输入创建新用户，已有用户名则恢复进度 |
| 浏览图鉴 | 画廊网格滚动 + 类别筛选 | 已解锁显示缩略图，未解锁显示剪影/锁 |
| 进入答题 | 点击未解锁条目 | 先展示知识，再进入答题 |
| 回答选择题 | 点击选项 | 选后立即判定 |
| 回答填空题 | 键盘输入 + 提交 | 支持模糊匹配（忽略大小写、空格） |
| 回答拖拽题 | 拖拽元素到目标区域 | 全部正确后判定通过 |
| 查看已解锁卡片 | 点击已解锁条目 | 全屏展示图鉴卡片详情 |
| 修改 ID | 点击头像/ID 区域 → 弹窗 | 修改后立即生效 |
| 选择头像 | 在个人设置中从已解锁图鉴选择 | 选中的图鉴插图裁剪为头像 |
| 分享 | 点击分享按钮 | 生成分享卡片，可截图保存 |
| 存档 | 自动 | 每次状态变化自动写入 localStorage |
| 读档 | 输入已有用户名 | 自动恢复该用户进度 |

### 答题形式规范

> 答题形式根据知识内容选择最合适的方式，同一个游戏中可混用多种形式。

| 题型 | 适用场景 | 交互方式 | 判定规则 |
|------|---------|---------|---------|
| 选择题 | 概念辨析、特征识别 | 点击 3-4 个选项之一 | 选对即通过 |
| 填空题 | 名称、术语、关键数据 | 键盘输入 | 模糊匹配（忽略大小写、前后空格、常见别名） |
| 拖拽匹配 | 分类、对应关系、排序 | 拖拽左侧元素到右侧目标 | 全部匹配正确即通过 |

> 每个条目只有 1 道题。题目形式在构建时确定，填入数据中。答错可无限重试，不惩罚。

### 交互反馈

| 操作 | 视觉反馈 | 状态变化 |
|------|---------|---------|
| 答对 | 解锁动画（卡片翻转 + 插图显现 + 庆祝光效）| entry.unlocked = true |
| 答错 | 温和提示 + 正确方向引导 + 重试按钮 | 无状态变化 |
| 解锁成就 | toast 通知滑入 | achievement.unlocked = true |
| 收集进度变化 | 顶栏进度数字更新动画 | progress +1 |
| 全部收集完成 | 跳转结局页 + 庆祝动画 | finished = true |

## 十、核心机制约束

> 以下是**不可更改的规则**，开发时必须遵守。

### ⚠️⚠️⚠️ 图鉴插图生成规则（最高优先级）

> **这是本项目最重要的约束，违反即判定为未完成。**
>
> **每一个图鉴条目必须拥有一张由 AI 生图模型生成的真实插图。绝对禁止用 CSS / SVG / Canvas / emoji / 文字描述替代。**

**什么是"生图"：**
- ✅ 调用生图模型（如图像生成 API / 工具），输入提示词，获得一张真实的图像文件
- ❌ 用 CSS 渐变、box-shadow、border-radius 画出来的图形 — **这不是生图**
- ❌ 用 SVG path 手动描绘的图形 — **这不是生图**
- ❌ 用 Canvas 代码绘制的图形 — **这不是生图**
- ❌ 用 emoji 或 Unicode 符号拼凑 — **这不是生图**
- ❌ 用纯文字描述"这里应该有一张图" — **这不是生图**
- ❌ 用 CSS 画一个"风格化的卡片"然后当作插图 — **这不是生图**

**构建时生成流程（强制执行）**：

1. **分析条目**：读取每个图鉴条目的知识描述，提取核心视觉特征
2. **编写提示词**：基于风格提示词基底 + 该条目的视觉特征，编写完整的生图提示词，提示词至少 30 词
3. **调用生图模型生成图片**：
   - 必须实际调用生图工具/API，获得图像数据
   - 如果当前环境有可用的生图工具，必须使用
   - 如果没有可用的生图工具，必须**停下来告知用户**，请求用户提供生图能力，**不可以用 CSS 画一个替代品然后继续**
4. **内嵌到游戏**：以 base64 编码嵌入 HTML 中

> **用户解锁图鉴后必须直接看到成品插图。** 生图是构建阶段的开发者职责，不是用户的操作。用户的体验是「答对 → 卡片翻转 → 精美插图已经在那里」，没有任何额外的等待或生成步骤。

**生图要求**：
- 所有插图必须使用同一套风格提示词基底，确保整本图鉴画风统一
- 插图应准确反映该条目的核心视觉特征
- 插图分辨率适合卡片展示（建议 512×512 或同等质量）
- **禁止跳过生图步骤**，禁止用纯色块、emoji、文字描述代替插图
- **禁止将生图任务留给用户**，所有插图必须在构建时就绑定到对应条目上

**如果无法生图怎么办**：
- **不可以**静默地用 CSS 画一个"替代品"然后当作完成了
- **必须**停下来，明确告诉用户："我需要生图能力来生成图鉴插图，但当前没有可用的生图工具。请提供生图工具/API，或者告诉我如何调用。"
- 在生图能力可用之前，可以先完成其他所有部分，但插图位置必须留空占位，标注"待生图"

### 答题解锁规则

- 每个条目绑定 1 道题，题目形式由内容决定
- 答对即解锁该条目的图鉴卡片
- 答错可无限重试，不惩罚、不记录错误次数
- 答错时提供温和的方向引导（不直接给答案）
- 答题前先展示该条目的核心知识（不是盲猜）
- 题目必须基于展示的知识内容出题

### 用户系统规则

- 进入游戏时必须输入用户名（ID）
- 用户名作为 localStorage 的 key 前缀，隔离不同用户的数据
- 用户可随时修改自己的 ID 显示名（不影响存储 key）
- 用户可从已解锁的图鉴插图中选择一张作为头像
- 默认头像为通用占位图标
- 无密码机制，任何人输入用户名即可进入对应进度

### 成就系统规则

> 成就是收集过程中的惊喜彩蛋，用来奖励特定的收集模式和里程碑。

**成就类型**：

| 类型 | 触发条件 | 风格要求 |
|------|---------|---------|
| 里程碑成就 | 收集达到特定数量（如 1/10/25/全部） | 正面鼓励，标记进度。如"初出茅庐"(1个)、"半途而废可不行"(半数)、"图鉴大师"(全部) |
| 类别成就 | 集齐某个类别的所有条目 | 与类别内容相关的趣味称号。如集齐鸟类→"观鸟达人" |
| 趣味成就 | 特定条目组合或特殊收集模式 | 彩蛋风格，奖励意外发现。如"连续答对 5 题"→"学霸附体" |
| 首次成就 | 第一次完成某类操作 | 引导性质。如"第一张图鉴"→"万里长征第一步" |

成就规则：
- 成就名称 6 字以内，描述 30-50 字
- 解锁时弹出 toast 通知（从顶部滑入，3s 后自动消失）
- 成就页展示所有成就：已解锁亮色，未解锁灰色 + 解锁条件提示
- 成就与称号系统联动——特定成就解锁特定称号

### 称号系统

- 玩家拥有一个当前称号，展示在个人信息和分享卡片中
- 称号通过成就解锁
- 初始称号为通用默认（如"见习收藏家"）
- 玩家可从已解锁的称号中选择一个展示
- 全图鉴收集完成后解锁最终称号

### 分享卡片规则

> 分享卡片是玩家对外展示收集成果的方式，必须精美到让人想截图发朋友圈。

分享卡片内容：
- 玩家头像（从已解锁图鉴选的）
- 玩家 ID
- 当前称号
- 收集进度（如 23/50）
- 装饰边框（与整体视觉风格一致）
- 游戏名称和 logo

分享卡片生成方式：
- 在页面内渲染为一个固定尺寸的区域（适合截图，如 375×667 或 1080×1920）
- 提供提示文字引导用户截图保存

### 结局页

- 全部图鉴解锁后触发
- 庆祝动画（仪式感强，如图鉴翻页浏览 + 金光 + 最终称号揭晓）
- 展示完整图鉴回顾（所有插图缩略图网格）
- 授予最终称号
- 展示所有已解锁成就
- 生成最终分享卡片
- 可返回画廊继续浏览

## 十三、开发指引

```
收到本 spec.md 后：
1. 进入 plan 模式，分析 SKILL 文件内容
2. 确定图鉴领域 → 确定视觉风格关键词 → 确定风格提示词基底
3. 提取所有图鉴条目，划分类别，设计成就
4. 根据风格提示词基底，为背景图编写提示词并调用生图模型生成
5. 为每个条目执行生图流程：
   a. 根据条目知识描述提取核心视觉特征
   b. 编写完整生图提示词（风格基底 + 条目特征）
   c. 调用生图模型生成插图，内嵌到游戏数据中
6. 为每个条目设计题目（根据内容选择最合适的题型）
7. 实现 UI 框架（画廊网格 + 卡片样式 + 答题界面）
7. 实现用户系统（用户名输入 + localStorage 绑定）
8. 实现答题 + 解锁 + 成就系统
9. 实现分享卡片 + 结局页
10. 遵守「核心机制约束」中的所有规则
11. 完成后执行全量自检（对照验收标准）
```
