# 通用人物传记模拟 spec.gameplay

## 玩法结构名称

通用人物传记模拟：从人物或作品输入中识别一个可体验的“人生主角”，生成一款已填充完成的传记模拟体验。玩家不是改写人物命运，而是在关键处做选择、听见真实/原作选择的反馈，并通过章末 CG、整体背景、主题化 UI 和粒子氛围进入人物的一生。

## 输出模式和设计粒度

| 项目 | 方案 |
|------|------|
| 输出模式 | `fixed-content` |
| 设计粒度 | 可填充玩法框架 |
| 最终用户体验 | 打开完成品后直接玩，不需要上传文件或配置生成参数 |
| 填充阶段 | 后续管线读取 `skills/内容文件`，识别人物、作品归属、章节、关系、决策、视觉风格和图片 prompt |

## 输入识别规则

### 输入类型

`skills/内容文件` 可能提供：

- 真实人物传记、年谱、访谈、新闻、回忆录、史料。
- 虚构人物资料、小说/剧本/漫画/游戏角色设定、IP 世界观资料。
- 单篇作品、作品集、诗篇、散文、评论、知识库摘录。
- 混合资料：人物资料 + 作品片段 + 时代背景 + 评论分析。

### 人物识别

1. 如果内容明确是人物传记或人物资料，直接识别该人物为体验主角。
2. 如果内容是作品，先判断作品中是否有明确且占绝对中心的主角。
3. 如果明确看得出作品主角是谁，体验主角 = 作品主角。
4. 如果作品主角模糊、多主角并列，或作品显然是诗篇、散文、随笔、论文、歌词、非叙事文本，则体验主角 = 作者 / 创作者。
5. 如果仍有多个候选，使用知识库检索和内容证据选择最适合做传记模拟的人物，并记录 `target_selection_reason`、`source_evidence`、`confidence`。

### 人物范围

不再限制为古风人物。可支持：

- 古代、近代、现代、当代人物。
- 中国、外国或跨文化人物。
- 真实人物、历史人物、公众人物、艺术家、科学家、创作者。
- 虚构人物、IP 角色、神话人物、传说人物、游戏/动漫/小说角色。

## 核心循环

```text
打开 app
→ 检查图片缓存
→ 首次缺图：主题化资产准备页生成图片并写入 IndexedDB
→ 图片 ready 后进入标题页
→ 人物介绍页：用一段简介说明 TA 是谁、年代/世界观、最出名的身份或事迹
→ 章节过渡
→ 叙事阅读 / 对话互动 / 决策选择
→ 反馈揭示人物真实选择或原作选择
→ 展示本章 CG 与章末感悟
→ 下一章
→ 结局评分 / 路线回顾 / 成就总结
```

核心规则仍保留旧版传记模拟的优点：选择提供代入和比较，不随意篡改人物真实人生或原作主线。玩家的选择主要触发“你为什么可能这样选”和“人物当时真正怎样选择 / 原作中怎样发生”的反馈。

## 内容槽位与 skills 映射

| 内容提取维度 | 映射到产品模块 |
|--------------|----------------|
| 人物身份、时代、地域、职业、关系网络 | `character_profile`、标题页、人物介绍页、视觉风格选择 |
| 人生阶段 / 作品剧情阶段 / 创作阶段 | `chapters` |
| 关键事件、冲突、转折、代价 | `decision` 场景和反馈 |
| 亲密关系、对立关系、导师/伙伴/敌手 | `dialogue` 场景、NPC 气泡、关系图 |
| 作品原文、名句、台词、引文 | 标题 quote、章末感悟、反馈引用 |
| 场景地点、时代物件、服饰、氛围 | 整体背景 prompt、章末 CG prompt、UI 材质 |
| 价值观与人物气质 | 结局评分、成就、视觉关键词、粒子类型 |
| 可验证事实 / 原作 canon | 准确性自检、引用来源、反馈事实校验 |

## 章节与内容规模

| 模块 | 默认范围 |
|------|----------|
| 章节数 | 5-7 章，默认 6 章 |
| 每章叙事节拍 | 必须完整呈现处境、前因与背景、事件苗头、关系/外部处境、事件推进、选择压力、玩家选择、真实/原作选择揭示、章末 CG、章末感悟 |
| 对话 / 互动 | 决策前至少 1 个多轮对话或互动单元，包含 2-3 轮推进 |
| 决策场景 | 每章 1 个主要决策 |
| 章末感悟 | 每章 1 个 |
| 成就 | 每个主要决策可关联 1 个 |
| 图片资产 | 1 张整体背景图 + 每章 1 张章末 CG |

章节不强制从“少年到晚年”开始。真实传记可按人生阶段推进；虚构人物可按原作剧情弧线推进；作者型作品可按创作背景、代表作品、人生转折和精神主题组织。

## 图片资产范围与生图时机

### 生图范围

必须真实生图，不允许 CSS/SVG/Canvas/emoji/文字占位冒充。

| 图片类型 | 数量 | 用途 | 必需性 |
|----------|------|------|--------|
| 整体背景图 | 1 张 | 全产品底层氛围背景，不得包含任何文字、标题、水印或可读字符 | 必需 |
| 章末 CG | 每章 1 张 | 只在玩家选择和真实/原作选择揭示后出现，表现该章章末情感高潮 | 必需 |

暂不默认生成结局 CG；如果最终 spec 想强化结局表现，可把结局 CG 作为 P1 可选资产。章末 CG 不得出现在章节开头、中段普通叙事或决策之前。

### 生图时机

`IMAGE_GENERATION_TIMING = first-run-cache`

首次打开 app 时先进入主题化资产准备页：

```text
读取 IMAGE_ASSET_MANIFEST
→ 查 IndexedDB
→ cache hit：读取 Blob 并解码
→ cache miss：调用真实生图能力生成缺失图片
→ 写入 IndexedDB
→ 所有必需图片 ready
→ 进入标题页和核心体验
```

后续刷新或再次打开只读缓存，不重新生图。必需图片未 ready 前不得进入核心体验。

## 视觉方向

视觉不再固定古风，也不固定羊皮纸或单一背景。视觉由人物/作品共同决定。

### 画风生成方法

后续管线应先分析：

- 人物/作品类型：真实传记、虚构人物、作者型作品、神话传说、现代公众人物等。
- 时代与文化：古代、现代都市、科幻、校园、战争、宫廷、舞台、实验室、游戏世界等。
- 情绪曲线：明亮、悲怆、荒诞、热血、孤独、浪漫、悬疑、史诗、冷静等。
- 媒介与质感：油画、胶片摄影、像素、赛博概念图、手绘插画、版画、3D、拼贴、档案视觉等。

每个项目生成一套统一 `STYLE_PROMPT_BASE`，所有图片共享同一画风基底，再为背景图和每章 CG 添加不同主体、场景、动作、光影和构图。

### UI 框体

UI 框体结构保持固定：标题页、资产准备页、章节过渡、叙事面板、对话气泡、决策面板、反馈页、章末感悟、结局页。

但 UI 设计必须随人物主题变化：

- 面板材质可从档案夹、玻璃 HUD、纸张、金属铭牌、舞台字幕框、漫画分镜、仪器面板、织物、石刻、游戏菜单等派生。
- 框体轮廓、边框、角标、按钮形态、进度条和 toast 需要主题化。
- 首页粒子必须更丰富，且与人物/作品匹配，例如花瓣、尘埃、代码雨、星尘、灰烬、舞台光尘、海雾、纸片、像素碎片、墨点扩散。
- 粒子数量默认 12-30 个，`pointer-events: none`，不得遮挡文字或按钮，必须支持 `prefers-reduced-motion`。

## 准确性模式

建议使用自适应准确性规则，并在最终 spec 中写入：

- 真实人物 / 历史人物 / 当代人物：`strict`，时间线、关系、称谓、事件和引用必须可验证。
- 虚构人物 / IP 角色：`semi-strict`，核心 canon、关系、剧情顺序和设定必须正确，允许叙事化表达。
- 神话、传说、民间角色：`semi-strict`，说明采用的版本来源，避免混淆不同版本。
- 原创或明确幻想向内容：`fantasy`，重在内部一致性，但仍必须忠实随附内容文件。

## 待确认点

1. 我已默认“每个 CG 一张”为“每章 1 张章末 CG”；CG 只在玩家选择和真实/原作选择揭示之后出现。
2. 结局 CG 暂不列为默认必需资产，只保留背景图 + 每章 CG。
3. 准确性模式建议采用上面的自适应规则，而不是单一 strict / semi-strict / fantasy。
4. 最终 spec 将新建在 `generated-specs/通用人物传记模拟/spec.md`，不直接覆盖旧的 `spec_古风人物.md`。
