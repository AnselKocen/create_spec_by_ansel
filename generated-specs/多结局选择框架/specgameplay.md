# 《多结局选择框架》玩法确认稿

> 本文件是 Spec Forge 流程中的 `spec.gameplay.md`，用于先确认玩法框架。确认后才生成最终 `spec.md`。
>
> 本稿只定义“可被不同 `skills/` 内容填充的结构玩法”，不预设具体世界观、题材、角色、地点、剧情或结局名称。所有具体内容都应在构建阶段从随附内容文件中读取、提取并设计。

## 1. 基础设定

| 维度 | 方案 |
|------|------|
| 玩法边界 | 最终用户直接玩完成后的多结局游戏，不包含上传、生成、换画风、重生成等工具流程 |
| 产品类型 | 分支叙事选择游戏 / 多结局文字冒险 |
| 产品名称 | 暂定《多结局选择框架》，最终可由内容文件主题替换为 `{{GAME_NAME}}` |
| 来源素材 | 随最终 spec 一起提供的完整内容文件，通常位于 `skills/` 文件夹 |
| 准确性模式 | 默认按内容类型决定：知识/真实人物/历史为 strict 或 semi-strict，虚构/幻想题材为 fantasy |
| 核心目标 | 玩家通过选择、失败、回看线索和重试，逐步抵达好结局 |
| 结局框架 | 至少 5 个结局，建议 5-8 个：多个坏结局 + 1 个好结局 + 可选隐藏结局 |
| 主要交互 | 阅读推进、选择分支、检查线索、触发结局、查看结局档案、重开探索 |
| 生图策略 | 固定生成 1 张真实 AI 封面图 + 每个结局 1 张真实 AI 结局图；不生成章节 CG、过程插图或 UI 贴图 |

## 2. 设计原则

这是一个“结构玩法”， 最终实现时，构建阶段必须读取随附 `skills/` 内容文件全文，并基于其中的内容设计：

- 游戏主题与世界观。
- 玩家身份与目标。
- 主要角色、事件、知识点或对象。
- 分支选择的语境和选项文本。
- 坏结局、好结局、隐藏结局的具体名称和触发条件。
- 线索、档案、成就、视觉意象和生图提示词。

本玩法只规定多结局游戏应如何运转：选择如何改变路线，坏结局如何反哺下一轮，好结局如何被推理出来，隐藏结局如何作为深度探索奖励出现。

## 3. 核心玩法循环

```text
进入内容驱动的章节 / 场景
→ 阅读当前处境、知识语境或事件冲突
→ 检查 0-3 个可选线索 / 信息点
→ 做出 2-3 选一的关键选择
→ 更新隐藏变量、路线标记和已知信息
→ 进入后续场景 / 坏结局 / 好结局 / 隐藏结局
→ 解锁结局档案和失败线索
→ 重开或从关键节点回退
→ 用上一轮信息逼近更优结局
```

坏结局不是单纯失败画面，而是学习机制。每个坏结局都必须向玩家揭示一个可用于下一轮的规则、因果、误区、人物动机、知识判断或路线信息。

## 4. 内容文件如何驱动游戏设计

任意 `skills` 内容都不应被强行套进固定剧情或固定题材。实现者必须先理解内容自身的对象、规则、冲突和价值，再设计与之匹配的分支玩法。

| 提取维度 | 需要回答的问题 | 转化为游戏设计 |
|----------|----------------|----------------|
| 核心对象 | 内容围绕什么人、事、物、概念、系统或关系展开？ | 玩家目标、章节主轴、关键角色 / 对象 / 概念 |
| 目标与欲望 | 内容中什么算达成目标，什么算偏离或失败？ | 主线目标、好结局条件、阶段任务 |
| 规则与约束 | 有哪些条件、边界、顺序、规则或因果不能违反？ | 隐藏变量、分支门槛、坏结局触发 |
| 冲突与取舍 | 有哪些两难、代价、矛盾、误判或价值冲突？ | 关键选择节点和选项设计 |
| 信息差 | 哪些信息适合通过探索、失败或档案逐步揭示？ | 线索系统、结局档案、坏结局反哺 |
| 常见错误 | 玩家或内容主体最容易在哪里理解错误或走错？ | 坏结局、错误路线、失败线索 |
| 关键转折 | 哪些内容会改变玩家对目标、规则或关系的理解？ | 中段反转、章节转折、隐藏路线入口 |
| 主题回收 | 内容最终希望玩家理解、记住或重新看见什么？ | 好结局、隐藏结局、结局总结 |
| 视觉符号 | 哪些物件、场景、材料、色彩或动作最能代表内容？ | UI 风格、SVG 装饰、粒子、生图资产 |
| 语气节奏 | 内容适合什么情绪、速度和叙述口吻？ | 文案语气、转场、按钮反馈、整体氛围 |

映射顺序：

1. 先提取核心对象和核心目标。
2. 再提取规则、约束、误区和取舍。
3. 将规则和误区转化为隐藏变量、选择后果和坏结局。
4. 让每个坏结局揭示一条下一轮可用的线索。
5. 让好结局回收内容的核心规则、主题或价值。
6. 若设置隐藏结局，它必须来自内容中的深层主题、反例、边界条件或特殊组合。

## 5. 章节结构框架

建议使用 4-7 个章节。章节名称和具体内容由 `skills` 提取后生成。

| 阶段 | 章节职责 | 设计要求 |
|------|----------|----------|
| 开场 | 建立主题、玩家身份、目标和第一个冲突 | 不能暴露所有规则，只给玩家一个可行动目标 |
| 学习段 1 | 引入第一类选择后果 | 至少可导向 1 个早期坏结局 |
| 学习段 2 | 引入更复杂的内容规则或人物动机 | 坏结局开始提供可复用线索 |
| 转折段 | 揭示核心误区或反转信息 | 玩家开始理解“怎样才能到好结局” |
| 收束段 | 综合考验前面学到的信息 | 多个变量共同决定结局 |
| 终局段 | 进入好结局、坏结局或隐藏结局 | 结局必须回收主要线索与核心主题 |

## 6. 结局框架

最终游戏至少包含 5 个结局。建议结构如下：

| 结局类型 | 数量 | 设计职责 |
|----------|------|----------|
| 坏结局 | 至少 3 个，推荐 4-6 个 | 展示错误理解、错误路线、错误取舍或核心误区的后果 |
| 普通 / 中性结局 | 可选 0-2 个 | 玩家达成部分目标，但留下代价或缺憾 |
| 好结局 | 必须 1 个 | 玩家理解核心规则、作出关键正确选择并完成主要目标 |
| 隐藏结局 | 推荐 1 个 | 玩家在理解好结局基础上完成额外探索、反向选择或深层条件 |

结局名称、叙事文本和触发条件必须来自具体内容，不得预先写死为固定题材。

### 坏结局设计规则

每个坏结局必须至少承担一个功能：

- 揭示一个隐藏规则。
- 暴露一个错误假设。
- 证明某个选择的代价。
- 提醒玩家某条线索很重要。
- 展示某个角色、概念或系统的真实边界。

坏结局页必须包含：

- 结局标题。
- 简短结局叙事。
- 本轮关键选择回顾。
- 解锁的失败线索。
- “重试 / 回到关键节点 / 查看档案”入口。

### 好结局设计规则

好结局不能靠随机选中，必须由前面坏结局或线索帮助玩家推理得到。

好结局通常要求：

- 至少理解 2-3 个核心规则或关键内容点。
- 避免最常见的错误路线。
- 在终局前做出符合主题和因果的选择。
- 回收主要内容文件中的核心价值、知识结论或叙事主题。

### 隐藏结局设计规则

隐藏结局不是简单“更好结局”，而是深度玩家奖励。

可选触发方式：

- 已达成若干坏结局并收集关键失败线索。
- 已满足好结局条件，但在终局选择一条反常路线。
- 收集隐藏物、读完可选档案或完成特殊组合。
- 对 `skills` 内容中的深层主题、反例、边界条件作出正确判断。

隐藏结局必须有独立意义，不应只是好结局换一句文案。

## 7. 隐藏变量框架

具体变量名称应由内容文件决定，但推荐保留 3-5 个隐藏变量，避免只有单一路线判断。

| 变量类别 | 用途 | 示例，不固定 |
----------|------|--------------|
| 理解度变量 | 判断玩家是否理解核心内容 / 规则 / 知识 | `understanding`、`insight`、`knowledge_integrity` |
| 风险变量 | 判断玩家是否持续走高风险或错误路线 | `risk`、`instability`、`danger_level` |
| 关系 / 共情变量 | 判断玩家与角色、对象或系统的关系 | `trust`、`empathy`、`alignment` |
| 资源 / 证据变量 | 判断玩家是否保留关键物品、线索或证明 | `evidence`、`resource`、`proof` |
| 循环 / 压力变量 | 判断玩家重复失败、忽略线索或触发惩罚的程度 | `loop_pressure`、`mistake_count` |

玩家不应直接看到数值。界面应通过文本、视觉反馈、档案变化和分支可用性间接反馈状态。

## 8. 选择设计规则

选项不能写成明显的“正确 / 错误”。每个关键选择都应体现一个真实取舍：

- 快速推进 vs 谨慎确认。
- 自保 vs 帮助他人。
- 遵守规则 vs 质疑规则。
- 短期收益 vs 长期代价。
- 表面安全 vs 深层风险。
- 直觉判断 vs 证据判断。

对于知识类或真实内容类游戏，错误选项必须对应真实常见误区，不能随机胡编。

## 9. 失败线索与结局档案

游戏必须提供结局档案系统，让坏结局成为可积累信息。

档案至少包含：

- 已解锁结局列表。
- 每个结局的标题、类型、达成时间。
- 本轮关键选择路径。
- 解锁的失败线索。
- 可回看的重要文本或证据。

档案不能直接给出完整攻略。它应帮助玩家推理，而不是替玩家选择。

## 10. 用户动线

```text
封面标题页（AI 封面图 + 开始 / 继续 / 档案）
→ 点击开始 / 继续
→ 进入动效
→ 内容驱动的开场章节 / checkpoint
→ 选择节点
→ 路线推进
→ 坏结局 / 普通结局 / 好结局 / 隐藏结局
→ 结局档案更新
→ 重试或回到关键节点
→ 逐步解锁更深路线
```

关键体验要求：

- 第一次失败后，必须出现结局档案入口。
- 已读文本允许加速或跳过，但首次阅读应完整呈现。
- 可从关键节点重试，避免重复劳动过重。
- 结局数量和已解锁比例应让玩家看到探索进度。

## 11. UI 与视觉框架

视觉主题必须由 `skills` 内容决定，不得固定为某一种题材。

最终 spec 应要求实现者从内容文件中提取视觉关键词，例如：

- 故事类：地点、时代、情绪、代表物、光影。
- 科学类：实验器材、图表、粒子、公式、材质。
- 传记类：时代背景、人物气质、历史物件、书信或档案。
- 图鉴类：纸张、标本、分类标签、自然纹理或收藏册。
- 人格类：抽象符号、色彩维度、角色剪影或人格徽章。

固定 UI 结构建议：

- 封面标题页：AI 生成封面图、主题化标题、开始 / 继续 / 结局档案。
- 游戏页：主叙事区、线索区、选择区、进度 / 章节提示。
- 选择反馈页：显示短期后果，不立即剧透最终变量。
- 结局页：AI 生成结局图、结局文本、关键路线、解锁线索、重试入口。
- 结局档案页：已解锁结局和失败线索。

UI 质量要求：

- 标题不能是普通 h1，必须主题化。
- 封面标题页必须显示真实 AI 生成封面图，不能使用 CSS / SVG / Canvas / emoji / 文字占位冒充。
- 点击开始 / 继续后必须播放进入动效，例如封面图轻微推近、视差、柔焦淡出、主题化遮罩过渡，随后开场章节入场。
- 按钮不能是默认样式，必须与内容主题一致。
- 结局卡、线索条目、进度标记要有明确视觉层级。
- 必须使用 SVG/CSS 做主题化图标、边框、进度或装饰。
- 必须有粒子 / 氛围层，类型由内容决定，数量约 10-30 个。
- 必须尊重 `prefers-reduced-motion`；减少动态效果时，进入动效降级为短透明度过渡。

## 12. 封面图与结局图生成策略

本框架固定需要真实生图，但生图范围必须严格收窄：**固定生成 1 张封面图 + 每个结局 1 张结局图**。除封面标题页与结局页外，不生成章节 CG、过程插图、线索图、档案图、按钮图或 UI 贴图。

必须生成的图片：

- 封面图 1 张，用于封面标题页。
- 每个结局 1 张结局图，用于对应结局页和解锁后的结局档案。
- 如果最终游戏有 5 个结局，则生成 1 张封面图 + 5 张结局图。
- 如果最终游戏有 8 个结局，则生成 1 张封面图 + 8 张结局图。

最终 spec 必须要求：

- 生图是构建闸门，不是可选增强。
- CSS、SVG、Canvas、emoji、文字占位都不是生图。
- 封面图和所有结局图必须共享 `STYLE_PROMPT_BASE`。
- 封面图和每个结局图必须有完整 prompt、negative prompt、比例和真实输出路径或 data URL。
- 必须建立 `IMAGE_ASSET_MANIFEST`。
- `IMAGE_ASSET_MANIFEST` 中必须有且只有 1 张 `cover_image`，每个结局必须有且只有 1 张 `ending_cg`。
- 无法生成图片时必须停止并告知用户，不能假装完成。

封面图设计要求：

- 封面图必须表现游戏主题、核心情绪和主要视觉符号，但不得剧透具体结局条件或隐藏路线。
- 封面图不得包含可读文字、logo、水印或伪 UI；标题、按钮和入口信息必须由 HTML/CSS 叠加实现。
- 封面图构图必须为标题和开始按钮预留安全区域，移动端裁切后仍能保留主体。
- 封面图需要支持进入动效，例如轻微推近、视差、柔焦淡出或主题化遮罩过渡。

结局图设计要求：

- 图像必须表现该结局的最终状态、情绪和核心视觉符号。
- 坏结局图应表达“错误理解 / 错误路线的后果”，但不要依赖血腥、低俗或纯惊吓。
- 好结局图应表达目标达成和主题回收。
- 隐藏结局图应表达深层主题或特殊路线的独立意义。
- 结局图是结局页视觉核心；普通游戏过程仍主要由文字、UI、SVG/CSS、粒子和档案系统承担。

## 13. 内容文件 / skills 契约预留

最终 `spec.md` 必须保留“涉及的 SKILL / 内容文件清单”章节，并直接写明：

> 完整内容文件会与本 spec 一起提供，通常位于项目根目录的 `skills/` 文件夹中。构建产品时，必须读取每个文件的完整内容，并从中提取本 spec 需要的内容。

如果当前没有实际文件，最终 spec 使用占位符：

| 内容文件路径 | 覆盖模块 | 在本产品中的用途 |
|-------------|----------|------------------|
| `{{CONTENT_FILE_PATH}}` | `{{COVERED_MODULE}}` | `{{USAGE_IN_PRODUCT}}` |

禁止事项：

- 不得只依赖索引摘要。
- 不得只读文件名或标题。
- 不得根据标题自行编造核心内容。
- 不得把一个固定题材强行套到所有内容文件上。

## 14. 需要用户确认的点

请确认这版玩法框架是否可以进入最终 `spec.md`：

1. 不预设具体题材，主题、角色、章节和结局都由 `skills` 内容驱动生成。
2. 玩法边界保持清晰，最终用户直接玩完成后的多结局游戏。
3. 结局要求写成框架：至少 5 个，包含多个坏结局、1 个好结局、可选隐藏结局。
4. 坏结局必须解锁可复用线索，帮助玩家逼近好结局。
5. 最终 spec 中保留内容文件契约、封面图 + 结局图生图策略、UI/SVG/粒子、数据 Schema 和验收标准。
