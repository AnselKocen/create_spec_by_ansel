# 固定 spec 骨架模块

每份最终 `spec.md` 都必须使用以下章节顺序。用户的自然语言描述决定玩法结构、内容槽位、映射规则、视觉方向和准确性要求，不改变骨架。

可以根据玩法结构增删章节内部的小节和字段，但不得删除这些一级章节，不得改变“随附内容文件 / SKILL 契约”和“生图策略与资产契约”的必备地位。

## 固定章节顺序

### 一、产品概览

- 产品名称、来源素材、设计粒度、玩法结构、产品类型、核心主题槽位、目标体验、交互方式、内容规模、准确性模式。
- 用表格写清楚本 spec 要做成什么，不写宣传文案。
- 产品概览表的前两行必须固定为：

| 维度 | 值 |
|------|-----|
| 产品名称 | {{GAME_NAME}} |
| 来源素材 | {{BOOK_NAME}} |

- 不得把 `产品名称` 改成 `BOOK_TITLE`、`项目名称`、`绘本标题` 等其它字段。
- 不得把 `来源素材` 改成 `输入内容来源`、`随附内容文件` 等其它字段；随附内容文件契约必须放在第二章单独说明。
- 产品概览表必须包含 `设计粒度` 行，默认值为 `可填充玩法框架`。
- 输出模式是 Spec Forge 管线内部信息，只能在 `spec.gameplay.md` 或内部规划中出现；最终 `spec.md` 不得出现 `fixed-content`、`generator-tool` 或 `输出模式` 表格行。
- 默认最终产品是已填充完成的体验：`spec.md + 随附内容文件 / skills` 会在后续管线中被填充为一个已经完成的产品，最终用户直接玩、读、测试、收集或学习。
- 只有用户明确要求最终产品本身是“生成器 / 工具 / 创作器 / 改编器 / 输入任意内容生成结果”时，最终 spec 才描述用户可操作的生成工具流程。
- 已填充完成的产品中，具体主题、角色、章节、题目、条目、结局和图片 prompt 优先写成槽位、规则、数量范围和 Schema，由随附内容文件填充。
- 用户可操作的生成工具中，可以把输入内容、配置参数、生成、预览、编辑、导出写成最终用户功能，但仍必须保留 UI 美化、粒子、SVG/CSS、生图策略、数据 Schema 和验收标准。

### 二、涉及的 SKILL / 内容文件清单

必须直接写明：

> 完整内容文件会与本 spec 一起提供，通常位于项目根目录的 `skills/` 文件夹中。构建产品时，必须读取每个文件的完整内容，并从中提取本 spec 需要的内容。

必须包含表格：

| 内容文件路径 | 覆盖模块 | 在本产品中的用途 |
|-------------|----------|------------------|
| `{{CONTENT_FILE_PATH}}` | `{{COVERED_MODULE}}` | `{{USAGE_IN_PRODUCT}}` |

必须说明内容使用验收重点：

- 实现者需要读取完整内容文件。
- 核心内容应能追溯到文件原文。
- 摘要、文件名和标题只能作为导航，不作为核心内容依据。

建议同时写清内容提取维度：

| 提取维度 | 在本 spec 中映射到 |
|----------|--------------------|
| `{{CONTENT_DIMENSION}}` | `{{MAPPED_GAME_MODULE}}` |

### 三、目标用户

- 用户画像、使用场景、设备偏好、知识水平或体验门槛。

### 四、产品目的

- 核心价值、成功标准。
- 成功标准必须能被验收，不要只写“好玩”“沉浸”。
- 核心价值应描述这种玩法框架如何把任意随附内容转化为可体验的游戏 / 产品结构。

### 五、视觉风格规范

- 整体气质、配色、排版、UI 面板、按钮、卡片、动效。
- 必须包含 UI 美化要求、SVG/CSS 技巧、粒子/氛围设计、`prefers-reduced-motion`。
- 视觉风格必须由产品主题和内容文件共同决定。
- 画风应写成可组合方法：媒介、情绪、文化/时代、材质、光影、构图、UI 质感、动效气质，而不是只从固定示例中选择。
- 如果用户要求高精度 SVG，或核心体验依赖 SVG/Canvas 可视化、互动模拟、参数探索、地图/图表/结构图/机制图，必须按 `specforge/modules/svg-css-visual-primitives.md` 写出专项绘制指南：可视化原语速查、SVG/CSS 质感配方、领域动效模式、标注与数据编码、交互态视觉暗示、常见画错点。不得只写“高精度 SVG”“分层可视化”“酷炫动效”等抽象要求。

### 六、生图策略与资产契约

- 本产品是否需要真实位图资产。
- 如果不需要，写清“不需要真实生图资产”的范围，并说明核心视觉由 SVG/Canvas/CSS/HTML 实现。
- 如果需要任何真实位图资产，必须包含完整生图契约：图片资产范围、什么是生图、什么不是生图、统一画风基底、每图 prompt、negative prompt、静态 `IMAGE_ASSET_MANIFEST`、运行时 `IMAGE_ASSET_RUNTIME_STATE`、IndexedDB 缓存记录、无法生图必须停下。
- 必须声明 `IMAGE_GENERATION_TIMING`：`build-time` / `first-run-cache` / `hybrid`。
- 默认使用 `first-run-cache`：最终 app 首次启动时显示主题化“正在生成图片 / 正在准备视觉资产”页面，生成缺失图片并写入 IndexedDB；之后刷新或再次打开直接读取 IndexedDB Blob、创建 object URL、加载并解码，不重新生图。
- `build-time`：仅当用户明确要求或少量核心图必须打开即显示时使用；图片在实现 / 构建阶段生成并嵌入或打包，首次打开后仍可作为 `seed_source` 写入 IndexedDB，后续优先读缓存。
- `hybrid`：核心图片构建期生成，次要或大量图片首次启动生成并缓存。
- 首次启动缓存策略必须写清：资产准备页、IndexedDB Blob 存储、`cache_key` 规则、`prompt_hash` / 版本失效规则、生成锁、失败重试、cache hit 不调用生图 API、必需图片 `ready = true` 前不得进入核心体验。
- 必须写清图片预生成与预加载闸门：所有 `required = true` 的图片在进入核心体验前必须完成 `generated + cached/cache_hit + loaded + decoded + ready`；核心体验阶段不得逐页生成、逐页请求或逐页等待 required 图片。
- 必须写清图片展示链路：从 IndexedDB 读取 Blob，并用 `URL.createObjectURL(blob)` 创建本次会话 object URL 后展示；object URL 只用于当前会话，不能长期存储。

### 七、功能清单

- 按 P0 / P1 写功能。
- 每个功能写清解决什么问题、输入、输出。
- 已填充完成的产品：功能清单写最终用户在完成品中实际操作的功能，例如开始、推进章节、选择、拖拽、探索、回看、存档、解锁、结局等。
- 如果功能依赖内容文件，写成可填充槽位和映射规则，例如“章节列表由 `{{CONTENT_PHASES}}` 生成”。
- 用户可操作的生成工具：可以写输入、解析、生成、预览、修改、重新生成、导出、历史记录等工具功能，但必须明确这些是工具的核心玩法 / 工作流。

### 八、用户动线

- 写核心路径和功能流转。
- 必须说明开始、推进、完成、回看/重试/存档等关键路径。
- 已填充完成的产品：用户动线必须从已经完成的产品开始，例如“标题页 → 章节主页 → 内容体验 → 结局总结”，不能从“上传文件 / 输入内容 / 点击生成”开始。
- 用户可操作的生成工具：用户动线可以从“输入内容 / 选择参数 / 生成 / 预览 / 导出”开始。
- 如果产品需要真实位图资产，用户动线默认需要包含“打开 app → 检查图片缓存 → 资产准备页 → cache hit 读取 IndexedDB Blob 并创建 object URL / cache miss 生成缺失图片、转 Blob、写入 IndexedDB、加载解码 → required 图片全部 ready → 进入核心体验”。这属于资产准备流程，不改变核心玩法结构。

### 九、信息结构

- 页面/状态/模块结构。
- 写清导航关系、页面内容和主要交互。
- 已填充完成的产品：页面结构描述最终体验界面，不设置面向最终用户的生成参数页。
- 用户可操作的生成工具：页面结构可以包含输入区、生成设置区、预览区、编辑区、导出区。

### 十、数据设计

- 存储方案、核心实体、实体关系、关键 Schema。
- Schema 字段必须服务本产品玩法，不照搬无关字段。
- 已填充完成的产品：必须区分“构建 / 填充阶段从内容文件提取出的产品数据”和“运行时用户进度数据”。前者用于生成完成品，后者用于 localStorage 等存档。
- 用户可操作的生成工具：可以包含用户输入、生成任务、生成结果、版本历史、导出记录等运行时数据。
- 如果产品需要真实位图资产，数据设计必须包含三层图片资产 Schema：静态 `IMAGE_ASSET_MANIFEST`（资产计划、prompt、`seed_source`、`cache_key`）、运行时 `IMAGE_ASSET_RUNTIME_STATE`（`generation_status`、`cache_status`、`cached_blob_ref`、`loaded`、`decoded`、`ready`）、IndexedDB 缓存记录（Blob、`cache_key`、`prompt_hash`、版本、时间戳、错误）。
- `cached_blob_ref` 只能指向 IndexedDB Blob 记录，不能保存普通文件路径或 object URL。
- localStorage 只能保存轻量进度、版本和状态，不得保存图片、base64、大 data URL、Blob 字符串或 object URL。

### 十一、交互设计

- 操作映射、状态变化、反馈方式。
- 必须写 hover/active/切换/错误/成功/解锁等反馈。

### 十二、核心机制约束

- 写本产品不可更改的玩法规则。
- 必须包含内容文件如何被提取和映射。
- 必须包含准确性模式要求：strict / semi-strict / fantasy。
- 必须包含产品形态约束：用自然语言说明哪些流程属于构建 / 填充阶段，哪些流程属于最终用户体验；不得暴露内部输出模式标签。
- 必须包含设计粒度约束：说明本 spec 是“可填充玩法框架”，并列出哪些内容由 `skills/内容文件` 填入。

### 十三、内容 / 章节概览

- 根据玩法结构写章节、关卡、图鉴条目、人格类型、故事节点、知识点或人生阶段。
- 必须说明内容规模、递进关系和每部分如何来自内容文件。
- 已填充完成的产品：内容 / 章节概览应描述随附内容文件会如何被填充进最终产品结构。
- 章节、关卡、故事节点、知识点等可以用占位字段表达，例如 `{{CHAPTER_TITLE}}`、`{{KEY_CONFLICT}}`、`{{VISUAL_THEME}}`、`{{QUESTION_SOURCE}}`。

### 十四、技术约束

- 技术栈、部署方式、离线能力、外部依赖、性能要求、响应式要求。

### 十五、开发指引

- 写给下一步实现模型的执行步骤。
- 必须包含：读取完整内容文件、执行生图策略、实现图片预生成 / 预加载闸门、实现核心玩法、执行自检。
- 已填充完成的产品：开发指引中的读取内容、提取内容、填充数据是构建阶段动作，不应暴露成最终用户界面；若采用默认 `first-run-cache` 生图，资产准备页属于运行时视觉资产准备流程，不是用户可操作的生成器玩法。
- 用户可操作的生成工具：开发指引必须说明工具运行时如何处理用户输入和生成结果，且不得在前端暴露 API key 或模型密钥。

### 十六、验收标准

- 功能验收。
- 随附内容文件验收。
- 生图验收。
- 首次启动图片缓存验收（所有需要真实位图资产的 spec 默认适用，除非用户明确选择纯 `build-time`）：首次缺图时显示资产准备页；图片生成后写入 IndexedDB；刷新或下次进入 cache hit 时只读取 IndexedDB Blob、创建 object URL、load/decode 且不调用生图 API；prompt 或版本变化时只重生成失效图片；required 图片运行时状态全部 ready 后才进入核心体验。
- 图片展示链路验收：`<img>`、CSS 背景和 Canvas 绘制应使用 IndexedDB Blob 派生的 object URL，或构建期真实图片 seed 写入 IndexedDB 后的 Blob；object URL 不得长期存储。
- 视觉验收。
- 高精度 SVG/CSS 可视化验收（触发时必须适用）：检查是否有领域可视化原语速查、质感配方、标注/图例/单位、交互态反馈、参数连续动画和常见画错点自检。
- 交互验收。
- 数据验收。
- 响应式验收。
- 准确性 / 一致性验收。
- 产品形态验收：检查最终 spec 是否把构建 / 填充阶段和最终用户体验职责分清；已填充完成的产品必须有内容槽位和最终体验链路；用户可操作的生成工具必须具备完整工具工作流。
- 设计粒度验收：检查 spec 是否主要定义玩法结构、内容槽位、映射规则和约束，而不是把可由 `skills` 填充的细节提前写死。

## 差异发生在哪里

不同 spec 的差异主要发生在：

- 玩法结构和目标体验。
- 核心玩法循环。
- 内容文件的提取维度和映射方式。
- 章节、关卡、图鉴、人格、故事节点、知识点等内容对象。
- 数据 Schema 的具体字段。
- 视觉生成方法、粒子类型、SVG/Canvas/CSS 表现方式。
- 准确性模式和自检重点。

所有 spec 稳定保留：

- 固定章节顺序。
- 随附内容文件 / SKILL 契约。
- 生图策略与资产契约。
- UI / SVG / 粒子 / 动效的质量要求。
- 玩法先确认，再生成最终 spec。
- 最终验收标准必须存在。
