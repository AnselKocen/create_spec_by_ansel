# Generator Tool Mode

`generator-tool` 只在用户明确要求“生成器 / 工具 / 创作器 / 改编器 / 输入任意内容生成结果”时使用。

适用场景：最终产品本身就是一个工具，最终用户可以输入内容、选择参数、生成结果、预览、编辑或导出。这里生成的 spec 仍然是工具结构框架：定义工具如何组织输入、生成、预览、修改和导出，而不是只写一个单次示例。

## 核心关系

```text
spec.md + skills/内容文件
→ 定义生成工具的规则、模板、风格、默认库、约束和验收
→ 实现成一个用户可操作的生成工具
→ 最终用户输入内容并生成结果
```

## 与 fixed-content 的区别

| 维度 | fixed-content | generator-tool |
|------|---------------|----------------|
| 默认性 | 默认模式 | 需要用户明确选择 |
| 最终用户 | 玩 / 读 / 测 / 收集已完成内容 | 输入内容并生成结果 |
| `skills` 作用 | 后台填充具体内容 | 生成规则、示例库、模板库、默认知识源、校验依据 |
| 用户动线 | 从成品入口开始 | 从输入 / 配置 / 生成任务开始 |
| 功能清单 | 成品体验功能 | 输入、参数、生成、预览、编辑、导出、历史记录 |
| 数据设计 | 已填好的内容数据 | 输入数据、生成任务、生成结果、历史记录 |

两种模式都可以是“可复用结构框架”。区别在于：`fixed-content` 的最终用户体验已完成内容，`generator-tool` 的最终用户操作生成流程。

## 允许出现的工具功能

可以写：

- 输入内容 / 上传文件 / 粘贴文本。
- 选择目标类型、年龄段、风格、难度、长度、输出格式。
- 生成、停止、重试、重新生成。
- 预览、编辑、导出。
- 历史记录、版本、复制结果。
- 运行时文本生成或生图调用。

但必须写清：

- API / 模型调用不能用前端硬编码密钥。
- 生图必须是真实生图，不能用 CSS/SVG/Canvas 冒充。
- 生成失败必须暴露原因，不能输出伪完成品。
- 用户输入内容需要校验、安全过滤和错误处理。

## UI 质量要求

工具模式也不能降级成普通表单页。仍必须包含：

- 主题化输入区和预览区。
- 高质量 UI 面板、按钮、状态反馈。
- SVG/CSS 装饰和图标。
- 粒子 / 氛围 / 生成状态动效。
- 清晰的空状态、加载态、失败态、成功态。
- 响应式和无障碍。

## 生图规则

如果工具会生成图片：

- 必须说明哪些图片由工具运行时生成，哪些图片作为工具自身 UI / 示例资产构建期或首次启动生成。
- 必须要求完整 prompt / negative prompt / 统一画风基底。
- 工具自身 UI / 示例资产默认使用 `first-run-cache`：首次启动检查 IndexedDB，缺图才生成并缓存，cache hit 不重新生图。
- 用户输入后生成的结果图按生成任务运行，但生成结果也必须有等效资产记录，并写入 IndexedDB / 历史记录；刷新后已有结果图不得无故重新生成。
- 必须保留静态 `IMAGE_ASSET_MANIFEST`、运行时 `IMAGE_ASSET_RUNTIME_STATE` 或等效生成结果记录。
- 只有用户明确要求时，才使用 `build-time` 或 `hybrid` 准备工具自身的视觉资产。
- CSS/SVG/Canvas/emoji/文字占位都不是生图。

## 验收标准

必须包含：

- 输入 → 生成 → 预览 → 使用/导出的链路可完整走通。
- 生成失败有明确错误状态。
- 生成结果符合 spec 的内容、安全、视觉和数据约束。
- UI、SVG/CSS、粒子、动效、生图、Schema、验收标准仍然完整。
