# 可填充玩法框架方法

Spec Forge 默认生成“可填充玩法框架 spec”：它定义主题、角色、章节和结局等内容如何被后续管线填入。

后续管线会把 `spec.md + skills/内容文件` 结合起来，填入具体主题、世界观、角色、章节、题目、条目、插图 prompt 和细节文本。当前 spec 的职责是定义：这些内容应该如何被提取、映射、组织、交互、表现和验收。

## 三层关系

```text
内容源层：skills/内容文件提供具体内容
玩法结构层：spec 定义内容如何变成可玩的结构
表现约束层：spec 定义 UI、生图、SVG、粒子、动效、数据和验收
```

## 先问玩法结构

参考 spec 是结构模式库，可以作为结构能力被组合和扩展。访谈时优先问用户想要哪种体验或玩法结构，例如：

- 分支叙事选择 / 多结局文字冒险
- 章节推进 / 关卡闯关
- 图鉴收集 / 条目解锁
- 参数探索 / 互动模拟
- 问答测试 / 维度判定
- 线索拼图 / 推理解谜
- 时间线推进 / 人生阶段模拟
- 卡牌组合 / 配方合成
- 地图节点探索 / 区域解锁
- 绘本翻页 / 轻互动阅读

这些只是启发词。用户可以提出任意新的玩法结构，agent 应根据内容特征组合或扩展结构。

## 内容提取维度

无论内容属于什么主题，先从以下维度理解它：

| 维度 | 要提取什么 | 可映射成什么 |
|------|------------|--------------|
| 对象 | 人物、地点、物品、生物、概念、工具、符号 | 角色、卡牌、图鉴条目、可点击对象 |
| 关系 | 因果、亲疏、对立、层级、演化、依赖 | 分支条件、线索网络、关系图、解锁条件 |
| 过程 | 步骤、阶段、时间线、实验流程、成长路径 | 章节、关卡、回合、任务链、故事页 |
| 规则 | 公式、定律、限制、禁忌、判断标准 | 模拟参数、题目规则、胜负条件、自检项 |
| 冲突 | 风险、误解、目标冲突、选择代价 | 决策点、失败反馈、多结局变量 |
| 感受 | 氛围、价值观、情绪曲线、审美关键词 | 画风、UI 质感、粒子、动效、音画节奏 |
| 可操作变量 | 可以被拖拽、选择、排序、组合、调参的内容 | 交互控件、探索阶段、解谜机制 |
| 可验证内容 | 事实、公式、出处、时间、人物关系、答案 | strict / semi-strict 自检和验收 |

## 玩法框架应该写什么

`spec.gameplay.md` 先确认结构，不急着填满具体内容。推荐包含：

- 玩法结构名称：如“分支叙事选择游戏 / 多结局文字冒险”。
- 内容来源如何进入玩法：从 `skills` 提取哪些维度，分别映射到什么模块。
- 核心循环：玩家每轮做什么，系统反馈什么，如何推进。
- 内容槽位：章节、角色、题目、条目、线索、结局、成就、图片等由哪些占位字段承载。
- 可变范围：数量、节奏、难度、章节规模、结局数量、题目数量等。
- 表现方向：画风生成方法、UI 氛围、粒子/动效类型，而不是固定几个示例。
- 准确性模式：哪些内容需要严格校验，哪些可以创作发挥。

## 最终 spec 的写法

最终 `spec.md` 应把主题细节写成“槽位 + 规则 + 示例格式”：

```text
{{CORE_THEME}}：由随附内容文件提炼出的核心主题
{{CHAPTER_TITLE}}：根据内容阶段或知识递进生成的章节标题
{{KEY_OBJECT}}：从内容文件中提取的关键对象
{{CHOICE_NODE}}：由冲突、因果或价值判断映射出的选择节点
{{ENDING_RULE}}：由玩家变量组合触发的结局规则
```

如果用户明确给定了必须固定的主题、角色或世界观，可以把这些固定项写入 spec；其余细节仍优先保持为可由内容文件填充的结构。

## 参考 spec 的使用方式

从已验证 spec 里提取的是“结构能力”：

- MBTI / 人格测试：维度、题目、结果类型、反馈语气。
- 恐怖叙事：氛围递进、线索、选择、结局变量。
- 物理 / 化学：可视化、参数探索、答题反馈、准确性自检。
- 图鉴：条目、分类、解锁、插图资产。
- 传记：时间线、阶段、人物关系、选择与回顾。

新的 spec 可以混合这些结构能力，形成新的玩法框架。
