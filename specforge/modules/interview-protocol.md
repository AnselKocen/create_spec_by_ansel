# Spec Forge 分轮访谈协议

读完 Spec Forge 规则后，先进入访谈模式。不要直接生成最终 `spec.md`。

如果用户只说“阅读 specforge”“按 specforge 流程”“先问我方向”等启动语，读完规则后直接开始第一轮问题。不要先输出规则摘要、执行清单或长篇说明。最多用一句话确认“我按 Spec Forge 来，先问第一轮”，然后立刻提问。

访谈目标是把用户的自然语言想法整理成可填充玩法框架：玩法结构、内容槽位、`skills/内容文件` 映射方式、生图范围、生图时机、视觉方向、准确性模式和硬约束。

## 访谈原则

- 分几轮问，每轮 3-5 个问题即可。
- 读完规则后直接问第一轮问题，不复述规则。
- 用户已经回答过的信息不要重复问。
- 用户不确定时，给 1-2 个建议方向，让用户选或直接继续自然语言描述。
- 不需要等待用户一次性提供全部信息；逐步收敛。
- 问题要围绕“生成 spec 需要什么”，不要像无关问卷。
- 先确认玩法结构，再写视觉和验收。
- 生图至少必须问三个问题：哪些部分生图、是否有必须构建期预置的例外、哪些图片是进入核心体验前必须 ready 的 required 资产。默认生图时机为 `first-run-cache`：首次打开显示资产准备页，生成缺失图片并写入 IndexedDB，之后刷新或再次打开直接读 IndexedDB Blob 缓存。
- 只要需要真实位图资产，玩法方案和最终 spec 都必须默认包含图片预生成 / 预加载闸门：required 图片进入核心体验前必须全部生成 / 缓存 / 加载 / 解码完成。
- 问清楚后先生成 `spec.gameplay.md`，让用户确认；用户确认后再生成最终 `spec.md`。

## 推荐轮次

### 第一轮：方向和产品形态

确认最基本方向：

1. 你想要什么玩法 / 产品结构？
2. 这是可填充成品框架，还是用户可操作的生成工具？
3. 随本 spec 一起进入后续管线的 `skills/内容文件` 大概会是什么内容类型？例如故事、知识点、图鉴条目、人物资料、题库、实验材料等。
4. 目标用户是谁，主要在什么场景使用？

### 第二轮：玩法结构和内容槽位

围绕玩法本身收敛：

1. 玩家每一轮主要做什么？
2. 内容文件里的信息主要填到哪里：角色、章节、题目、条目、线索、结局、插图、参数、成就？
3. 结构是线性、分支、探索、收集、模拟、循环，还是混合？
4. 内容规模大概是多少：章节数、题量、条目数、结局数、图片数等。

### 第三轮：生图和视觉

确认视觉资产策略：

1. 哪些部分需要真实生图？
2. 默认使用 `first-run-cache`：首次打开生成并缓存，之后不再生图。有没有少数图片必须 `build-time` 预置，或需要 `hybrid`？
3. 哪些图片是进入核心体验前必须 ready 的 required 图片，哪些如果失败也不阻塞核心体验的 optional 图片？
4. 有无画风 / UI 氛围偏好？
5. 如果产品要求高精度 SVG / Canvas / 互动模拟，需要确认哪些对象必须精细绘制、哪些过程需要粒子/连续动画、是否要达到物理/化学参考 spec 那种绘制手册级指导。
6. 如果用户没给画风，按内容和玩法推导媒介、情绪、材质、光影、构图和动效气质。

### 第四轮：准确性和硬约束

确认实现边界：

1. 准确性模式：`strict`、`semi-strict`、还是 `fantasy`？
2. 哪些内容必须严格来自 `skills/内容文件`？
3. 哪些地方允许创作发挥？
4. 还有没有必须保留的规则、风格、禁忌、交付形式或技术要求？

## 生成玩法方案

信息足够后，先生成：

```text
generated-specs/<project-name>/spec.gameplay.md
```

玩法方案应包含：

- 玩法结构名称。
- 输出模式和设计粒度。
- 核心循环。
- 内容槽位和 `skills` 映射。
- 页面 / 状态粗结构。
- 生图资产范围、生图时机、required / optional 图片划分；默认写 `first-run-cache`，除非用户明确要求 `build-time` / `hybrid`。
- 图片预生成 / 预加载闸门：required 图片全部 ready 前不得进入核心体验；cache hit 只读 IndexedDB Blob 并创建 object URL；cache miss 生成缺失图片、转 Blob、写入 IndexedDB、load/decode。
- 视觉方向。
- 如果需要高精度 SVG / Canvas，写清最终 spec 将包含可视化原语速查、SVG/CSS 质感配方、动效模式、标注/图例/交互态和常见画错点。
- 准确性模式。
- 待确认点。

生成后请用户确认。用户提出修改时，先修改 `spec.gameplay.md`。

## 生成最终 spec

只有在用户确认玩法方案后，才生成：

```text
generated-specs/<project-name>/spec.md
```

最终 spec 必须遵守固定骨架、内容文件契约、生图契约、UI/SVG/粒子约束、数据 Schema 和验收标准。
