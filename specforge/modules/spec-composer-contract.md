# Spec Forge Composer Contract

生成最终 `spec.md` 时必须遵守：

0. **输出文件必须有项目名目录**
   - 生成文件落盘时，必须保存到 `generated-specs/<project-name>/`。
   - `<project-name>` 来自用户指定的 spec / 产品名称；如果用户未指定，必须根据玩法结构、产品标题或核心主题推断一个短而明确的名称。
   - 玩法方案保存为 `generated-specs/<project-name>/spec.gameplay.md`。
   - 最终 spec 保存为 `generated-specs/<project-name>/spec.md`。
   - 如果同名目录已存在，不得静默覆盖；必须询问用户，或另取版本名。

1. **必须先完成分轮访谈**
   - 生成任何文件前，先遵守 `specforge/modules/interview-protocol.md`。
   - 读完规则后直接开始第一轮问题，不输出规则摘要或执行清单。
   - 访谈分轮进行：方向和产品形态、玩法结构和内容槽位、生图和视觉、准确性和硬约束。
   - 每轮通常问 3-5 个问题；用户已经回答的信息不重复问。
   - 信息足够后，先生成 `spec.gameplay.md`；用户确认玩法方案后，才生成最终 `spec.md`。

2. **最终 spec 必须包含随附内容文件契约**
   - 每一份最终 `spec.md` 都必须包含一个明确的小节，推荐标题为“涉及的 SKILL / 内容文件清单”。
   - 不要把这组输入叫做“素材”。它是随最终 spec 一起交给实现模型的内容源文件，不是可选参考资料。
   - 小节中必须直接写明这个前提：**完整内容文件会与本 spec 一起提供，通常位于项目根目录的 `skills/` 文件夹中。**
   - 小节中必须对实现者说清楚：构建产品时必须读取每个文件的完整内容，并从中提取本 spec 需要的内容，映射到游戏数据、题目、章节、叙事、图鉴条目、人物阶段、可视化或机制设计中。
   - 必须包含表格：
     | 内容文件路径 | 覆盖模块 | 在本产品中的用途 |
     |-------------|----------|------------------|
     | `{{CONTENT_FILE_PATH}}` | `{{COVERED_MODULE}}` | `{{USAGE_IN_PRODUCT}}` |
   - 验收时重点检查：实现者是否读取了完整内容文件，是否能说明核心内容与文件原文的对应关系。
   - 如果生成 spec 时还没有实际内容文件，也必须保留该契约和占位符，供下一步实现时替换。
   - 内容文件如何进入产品必须随玩法结构和内容特征变化。优先提取对象、关系、过程、规则、冲突、感受、可操作变量和可验证内容，再映射成角色、关卡、问题、条目、线索、参数、结局、图像提示词或成就。

3. **输出模式只作为管线内部信息**
   - `fixed-content` / `generator-tool` 是 Spec Forge 的内部输出模式，可在 `spec.gameplay.md`、访谈记录或内部规划中出现。
   - 最终 `spec.md` 不得在产品概览中声明 `输出模式`，也不得直接写出 `fixed-content` 或 `generator-tool`。
   - 默认内部模式为 `fixed-content`。
   - 只有当用户明确要求“生成器 / 工具 / 创作器 / 改编器 / 输入任意内容生成结果”时，内部模式才使用 `generator-tool`。
   - 默认完成品模式必须遵守 `specforge/modules/fixed-content-mode.md` 的职责边界，但最终 spec 用自然语言描述“最终用户打开已填充完成的产品，内容提取和填充属于构建阶段”。
   - 用户可操作的生成工具必须遵守 `specforge/modules/generator-tool-mode.md` 的职责边界，但最终 spec 用自然语言描述输入、生成、预览、编辑和导出流程。
   - 两种内部模式都必须保留 UI 美化、SVG/CSS、粒子/氛围、生图策略、内容文件契约、数据 Schema 和验收标准。

4. **最终 spec 默认是可填充玩法框架**
   - 除非用户明确要求一次性具体实例，最终 `spec.md` 应描述一个可由 `skills/内容文件` 填充的玩法 / 产品结构。
   - 具体主题、世界观、角色、章节、题目、图鉴条目、结局、图片 prompt 等，优先写成槽位、数量范围、映射规则和 Schema。
   - 如果用户给定了必须固定的内容，可以把固定项写入 spec；其余内容仍保持可填充。
   - 产品概览表必须包含 `| 设计粒度 | 可填充玩法框架 |`，或在用户明确要求时写成 `| 设计粒度 | 具体实例 |`。
   - 玩法方案 `spec.gameplay.md` 应先确认结构：核心循环、内容槽位、提取维度、映射关系、用户体验路径、视觉策略和准确性模式。

5. **最终 spec 使用固定骨架**
   - 自然语言访谈只决定“要做什么产品、怎么玩、内容如何映射、视觉是什么气质”，不改变最终 spec 的核心结构。
   - 最终 `spec.md` 必须遵守 `specforge/modules/fixed-spec-skeleton.md` 中定义的章节顺序。
   - 产品概览表的前两行必须固定为 `| 产品名称 | {{GAME_NAME}} |` 和 `| 来源素材 | {{BOOK_NAME}} |`。
   - 产品概览表不得包含 `输出模式` 行；内部输出模式只保留在 `spec.gameplay.md` 或管线元信息中。
   - 产品概览表必须包含 `| 设计粒度 | 可填充玩法框架 |`，以提醒后续模型这是结构卷而不是锁死细节的单个产品。
   - 每一份最终 `spec.md` 都必须保留稳定的大结构：产品概览、随附内容文件 / SKILL 清单、目标用户、产品目的、视觉风格规范、生图策略与资产契约、功能清单、用户动线、信息结构、数据设计、交互设计、核心机制约束、内容/章节概览、技术约束、开发指引、验收标准。
   - 生图策略与资产契约、随附内容文件 / SKILL 清单、UI / SVG / 粒子 / 动效约束、数据 Schema、验收标准是固定必备章节，玩法结构变化时也要保留。
   - 如果某个产品明确不需要真实生图，也必须保留“生图策略与资产契约”章节，写清“不需要真实生图资产”的范围；一旦需要任何位图资产，必须执行完整生图契约。

6. **参考 spec 是结构模式库**
   - 已验证 spec 用于学习“结构能力”：核心机制如何描述、内容文件如何映射、视觉约束如何写、数据 Schema 如何落地、验收标准如何拆分。
   - 可以混合借用多个参考 spec 的结构能力，例如分支叙事 + 图鉴收集 + 成就系统，或参数模拟 + 问答反馈 + 粒子氛围。
   - 新 spec 的产品主题、玩法结构、内容槽位、数据字段、映射规则和视觉气质可以自由扩展，核心章节职责保持稳定。

7. **先锁玩法结构，再写视觉**
   - 最终 spec 必须体现已确认的“玩法方案”。
   - 玩法方案应优先确认结构：玩家循环、内容槽位、章节/节点规则、结局或反馈机制、数据状态和验收方式。
   - 玩法方案中如果需要真实图片，必须确认图片资产范围：哪些模块需要生图、每类图片大致数量、是否为进入核心体验前必需、是否允许作为可选资产延迟生成。
   - 玩法方案中必须声明生图时机；默认使用 `first-run-cache`。只有用户明确要求构建期预置核心图时，才使用 `build-time` 或 `hybrid`。
   - 视觉部分服务玩法结构和内容文件，不先用固定画风例子限制主题。

8. **必须保留的可复用强约束**
   - 随附内容文件限制：必须读取完整 `skills/` 或内容文件，必须写清内容文件如何映射进产品，并让核心内容可追溯到文件原文。
   - 生图限制：哪些部分生图、什么是生图、什么不是生图、必须提示词、统一画风基底、生图资产闸门、静态 `IMAGE_ASSET_MANIFEST`、运行时 `IMAGE_ASSET_RUNTIME_STATE`、IndexedDB 缓存记录、`IMAGE_GENERATION_TIMING`、无法生图必须停下。
   - 首次启动生图缓存：凡是需要真实位图资产，默认最终 spec 必须要求资产准备页、IndexedDB Blob 缓存、cache key / prompt hash / 版本失效、生成锁、失败重试、cache hit 不重新生图和缓存验收。只有用户明确选择纯 `build-time` 时，才可以不把资产准备页作为首屏流程，但仍要有静态资产清单和真实生图验收。
   - UI 美化：标题不是普通 h1，按钮/卡片/面板必须主题化。
   - SVG/CSS 技巧：用 SVG 做装饰、图标、边框、进度，CSS 做状态和动效。
   - 高精度 SVG / CSS 可视化：如果用户要求高精度 SVG，或产品核心依赖 SVG/Canvas 互动模拟、参数探索、地图、图表、结构图、机制图，最终 spec 必须按 `specforge/modules/svg-css-visual-primitives.md` 写到绘制手册级别，包含可视化原语速查、SVG/CSS 质感配方、领域动效模式、标注与数据编码、交互态视觉暗示、常见画错点和专项验收。不能只写“高精度”“分层”“酷炫”。
   - 粒子/氛围：根据主题选择粒子类型，数量、性能、`prefers-reduced-motion` 要写清楚。
   - 动效与验收：屏幕切换不能瞬切，hover/active 有反馈，验收标准要能约束实现。

9. **不要写成宣传文案**
   - spec 是给下一步大模型/编码助手执行的设计文档。
   - 要写清输入、输出、状态变化、数据结构、验收条件。

10. **保留强约束措辞**
   - 诸如“最高优先级”“必须实现”“不可更改”“不通过则判定未完成”“什么不是生图”等内容不能弱化。
   - 每份最终 spec 都必须保留“生图策略与资产契约”章节。如果本产品不需要真实位图资产，必须在该章节中明确写清“不需要真实生图资产”的范围；如果需要任何真实位图资产，必须完整插入生图契约。
   - 如果产品需要真实位图资产，默认必须把资产准备页写成进入核心体验前的视觉资产准备流程，不能把它写成普通生成器玩法。
   - 如果用户选择需要生图，不能提供 CSS/SVG/Canvas 占位图作为替代方案，也不能把“待生图”标记为通过验收。

11. **输出完整 Markdown**
   - 只输出最终 spec Markdown 正文。
   - 不输出解释、寒暄、代码块包裹或额外分析。
