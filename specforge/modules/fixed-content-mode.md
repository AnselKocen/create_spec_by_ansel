# Fixed Content Mode

`fixed-content` 是默认的内部输出模式，对外可以理解为“可填充成品框架”。

适用场景：用户想要一个游戏、绘本、图鉴、测试、互动学习产品、传记体验、叙事作品等最终成品。Spec Forge 先生成可填充的玩法结构，后续管线再把 `skills/内容文件` 填入这个结构并实现成最终产品。

## 核心关系

```text
可填充玩法框架 spec
+ skills/内容文件
→ 后台填充管线填入主题、章节、题目、故事、条目、角色、画风、提示词
→ 实现管线做成最终 HTML 产品
→ 最终用户直接玩 / 读 / 测 / 收集 / 学习
```

## 写作原则

- 最终 spec 像一份“填空试卷 + 强约束”：它规定内容槽位、映射规则、交互方式和验收标准。
- `skills/内容文件` 提供具体主题和细节；spec 负责说明这些内容如何进入章节、题目、角色、图鉴、剧情、可视化或机制。
- 功能清单、用户动线、信息结构描述最终用户体验；内容提取、自检等管线动作放在内容文件契约、开发指引和验收标准里。只要需要真实位图资产，默认生图是首次启动资产准备流程：先查 IndexedDB，缺图才生成并缓存，cache hit 后直接进入体验；这不改变核心玩法结构。
- 当某个细节依赖内容文件时，用占位符和映射规则表达，例如 `{{CORE_THEME}}`、`{{CHAPTER_TITLE}}`、`{{KEY_OBJECT}}`、`{{ENDING_RULE}}`。

## 功能清单应该写什么

写最终用户能体验到的功能，例如：

- 开始游戏 / 开始阅读 / 开始测试。
- 章节推进、翻页、答题、探索、收集、解谜、回看。
- 成就、结局、进度保存、分享卡片。
- 轻互动、反馈动画、视觉状态。
- 首次启动资产准备页（需要真实位图资产时默认存在）：检查 IndexedDB 缓存、生成缺失图片、显示进度、写入缓存、完成后进入核心体验；刷新或再次打开 cache hit 时不重新生图。

与内容填充相关的动作通常写到这些章节里：

- 涉及的 SKILL / 内容文件清单。
- 核心机制约束。
- 生图策略与资产契约。
- 开发指引。
- 验收标准。

## 用户动线

推荐从最终体验入口写起：

```text
标题页 → 开始 → 章节/页面/题目/关卡 → 反馈/收集/成就 → 结尾/总结
```

如果某个页面内容由 `skills` 填充，在动线中写出页面职责和槽位即可。

## 数据设计

可以保留填充产物和来源追踪，例如：

- `source_material`
- `content_files`
- `skill_mapping`
- `content_slots`
- 静态 `IMAGE_ASSET_MANIFEST`
- 运行时 `IMAGE_ASSET_RUNTIME_STATE`
- IndexedDB 图片缓存记录
- `safety_report`

同时区分运行时用户状态，例如：

- `progress`
- `unlocked_items`
- `first_answers`
- `ending_flags`

## 验收标准

必须包含：

- spec 明确说明内容如何从 `skills/内容文件` 映射进产品结构。
- 最终产品经填充后可以直接体验完整内容。
- 内容槽位、数据 Schema、图片资产和验收项能支持不同内容文件复用。
- UI、SVG/CSS、粒子、动效、生图、Schema、验收标准仍然完整。
