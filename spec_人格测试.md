## 二、维度设计规则

> **维度是整个游戏的骨架。** 4 个维度必须从 skills/书的主题、人物、世界观中提炼出来，不可照搬 MBTI。

### 维度提炼流程（强制）

1. **读取所有 skills 内容**，理解书的核心主题、人物性格、世界观设定
2. **提炼 4 个对立维度**，每个维度代表书中一种核心的价值观/行为模式/认知方式的对立
3. **为每个维度命名**：用书中的意象或概念命名两极，不用通用心理学术语
4. **为每个维度设计字母代号**：每极 1 个字母，4 维度 = 8 个字母，人格代号由 4 个字母组合

### 维度设计约束

- 4 个维度之间必须相互独立——不能一个维度是另一个的子集
- 每个维度的两极必须是真正的对立，不是好/坏之分——两极都应该有各自的魅力
- 维度命名必须贴合书的气质，让读过书的人会心一笑
- 禁止使用 MBTI 的原始维度名（外向/内向、感觉/直觉、思考/情感、判断/知觉）

## 三、人格类型设计规则

> 4 个维度 × 2 极 = 最多 16 种组合。从中选取 10-16 种作为正式人格类型。

### 人格设计约束

- 每种人格有一个 **代号**（可以是 4 字母组合，也可以是来自 skills/书中的意象、术语、符号等，由开发者根据主题自行决定代号风格）
- 每种人格有一个 **人格名称**（2-6 字，贴合主题气质，可用书中意象、skills 中的概念、或自行创造）
- 每种人格有一段 **核心描述**（80-150 字）
- 每种人格有 **优势与盲区**（各 2-3 条）
- 每种人格标注 **最接近书中哪个角色/概念**（如果 skills 中有对应的概念也可以用）
- 人格命名风格统一——不要混搭，选定一种命名方式后所有人格保持一致

### 人格数据格式

```json
{
  "id": "ABCD",
  "name": "人格名称",
  "description": "核心描述（80-150 字）",
  "strengths": ["优势1", "优势2"],
  "blind_spots": ["盲区1", "盲区2"],
  "closest_character": "最接近书中的角色/概念",
  "character_reason": "为什么接近（1-2 句）"
}
```

## 四、题目设计规则

> **题目质量决定游戏好不好玩。** 35 道题必须有趣、诙谐、贴合书的世界观，不能是通用心理测试的翻版。

### 题目设计约束

- 共 35 道题，均匀覆盖 4 个维度（每个维度 8-9 道题）
- 每道题恰好 2 个选项（A/B），分别对应该维度的两极
- **没有对错之分**——两个选项都是合理的、有吸引力的
- 题目必须使用书中的场景、人物、概念作为素材——不是"你更喜欢独处还是社交"，而是用书中的具体情境来提问
- **语气诙谐有趣**——像朋友之间的趣味测试，不是正经心理量表
- 选项文字简短有力（10-25 字），不写长段落
- 题目之间要有变化——不能 8 道题问的都是同一个套路

### ⚠️ 反两极分化约束（强制）

> **核心原则：玩家做完测试后应该觉得"我也不确定自己会被分到哪种人格"，而不是"我做到第 5 题就知道自己是什么类型了"。**
>
> 如果玩家能一眼看出选项在测什么维度、哪个选项对应哪一极，题目就失败了。

**必须遵守的规则**：

1. **情境优先，倾向隐藏**——每道题必须包裹在一个具体的书中情境里，玩家的注意力应该在"这个情境下我会怎么做"，而不是"我是 A 类人还是 B 类人"。两个选项应该像是两种都合理的行动方案，而非两种人格标签。

2. **禁止对称表述**——两个选项不可以用明显对立的措辞（如"冒险 vs 谨慎"、"独处 vs 社交"、"理性 vs 感性"）。如果把两个选项的关键词提出来放在一起，不应该一眼看出是反义词。

3. **选项要有具体行动**——选项写的是"你会做什么"，不是"你是什么样的人"。
   - ❌ `"我喜欢提前规划"` — 这是自我申报
   - ✅ `"先把地图摊开，标出所有补给点再出发"` — 这是行动

4. **至少 5 道反向题**——35 道题中至少 5 道，选项 A 对应的不是该维度的"第一极"而是"第二极"（即 A/B 与极的对应关系反转）。这防止玩家发现"一直选 A 就是某种人格"的规律。

5. **选项吸引力均衡**——两个选项应该让 50% 左右的人选 A、50% 选 B。如果一个选项明显更"正确"、更"酷"、更"成熟"，就会导致分布偏斜。避免一个选项是积极正面的而另一个消极负面的。

6. **同一维度的题目要用不同角度切入**——测同一个维度的 8-9 道题，不能反复用相似的情境和措辞。每道题应从不同的角度探测同一个倾向（如行为偏好、价值判断、面对压力的反应、社交场景下的选择等）。

### 题目风格示例

**好的题目**（情境具体，倾向隐蔽）：

> 🎯 测试维度：理性-感性（但玩家看不出来）
>
> 你意外翻到一封 [书中角色] 留下的旧信，信里提到一个你从没听说过的地点。你会——
> A. 「去把信里提到的那条街走一遍，也许能碰到点什么」
> B. 「先查查这个地名在书里其他地方有没有出现过」
>
> ✅ 两个选项都是合理行动，没有对错
> ✅ 表面上是"探索方式"的选择，实际上在测理性-感性
> ✅ 不存在"明显更好"的选项

**坏的题目**（两极分化，一眼看穿）：

> ❌ 面对未知，你更相信——
> A. 直觉和感受
> B. 数据和分析
>
> 问题：直接用了"直觉 vs 数据"这种经典对立，玩家立刻知道在测什么

**坏的题目**（一个选项明显更"好"）：

> ❌ 朋友遇到困难时，你会——
> A. 认真倾听，给予情感支持
> B. 不太在意，觉得他们自己能解决
>
> 问题：A 明显更"善良"，导致绝大多数人选 A，分布严重偏斜

## 五、计分与匹配逻辑

### 计分规则

- 每道题选 A 或 B，对应的维度极 +1 分
- 答完 35 题后，每个维度计算两极的比例（如维度 1：极A 得 6 分，极B 得 3 分 → 极A 占 67%）
- 每个维度取得分更高的一极，组合成人格代号

### 维度倾向条

- 结果页展示 4 个维度的倾向条（进度条样式）
- 显示百分比，如「极A 67% ←→ 极B 33%」
- 倾向越极端，颜色越深/饱和度越高

### 人格匹配

- 代号直接映射到对应的人格类型
- 如果某个组合没有对应人格（选了 10 种而非 16 种），映射到最接近的人格

## 六、兼容性系统

> **答完测试后，玩家最想知道的就是"我跟谁合得来"。**

### 兼容性规则

- 基于维度计算，不需要手写 120 种两两配对
- 每种人格展示 2-3 种关系类型，具体关系类型由开发者根据主题自行设计（如最佳拍档/天生对手、灵魂共鸣/命运宿敌等），命名贴合主题气质

### 兼容性数据格式

> 关系类型的字段名和数量由开发者根据主题自行设计。以下为示例结构：

```json
{
  "personality_id": "人格代号",
  "relationships": [
    { "type": "关系类型名称（贴合主题）", "target_id": "对应人格代号", "reason": "原因描述" }
  ]
}
```

## 七、视觉风格规范

> 人格测试类游戏的视觉必须让人想截图分享。结果页是核心——必须精美到社交媒体级别。

### 整体气质（由素材主题决定）

> 开发者根据 skills/书的主题确定视觉风格。配色、字体、装饰元素都应该从主题中来。

**风格确定流程（强制）**：
1. 分析书的主题和气质
2. 从 15 种 UI 风格套餐中选择最匹配的一套——选定后底色、维度色、卡片、按钮、进度条、字体、粒子全部绑定
3. 所有 UI 围绕所选风格套餐设计，不可跨风格混搭
4. 确定人格形象的画风

### UI 风格套餐（选一套）

> **从下方 15 种风格中选择一套**，选定后该风格的底色、维度色、卡片、按钮、进度条、字体、粒子全部绑定使用，不可跨风格混搭。4 个维度必须各有独立代表色，色相差异明显，在答题页和结果页倾向条上可清晰区分。维度色在答题页也应体现——当前题目属于哪个维度，该维度色可用于卡片边框、选项底色或进度条高亮。

**通用约束**：
- 底色必须为浅色调（亮底深字），不使用深色/暗色背景
- 字体通过 CDN 加载，必须有系统字体 fallback，不使用浏览器默认 serif / sans-serif
- 粒子数量 10-20 个，全程显示（首页+答题+结果页），不干扰答题操作，尊重 `prefers-reduced-motion`
- 结果页配色要适合截图分享——社交媒体上要好看

**人格形象提示词模板**（选定画风后固定基础部分）：

```
固定基底：{{CHARACTER_STYLE_BASE}}
每种人格追加：[服装颜色/配饰/手持物/表情/姿态，来自该人格的特征和书中意象]
```

> 所有人格形象共享同一个基底提示词，只替换角色细节。保证画风统一，只是角色不同。

**什么是"生图"：**
- ✅ 调用生图模型（如图像生成 API / 工具），输入提示词，获得一张真实的图像文件
- ❌ 用 CSS 渐变、box-shadow、border-radius 画出来的图形 — **这不是生图**
- ❌ 用 SVG path 手动描绘的图形 — **这不是生图**
- ❌ 用 Canvas 代码绘制的图形 — **这不是生图**
- ❌ 用 emoji 或 Unicode 符号拼凑 — **这不是生图**
- ❌ 用纯文字描述"这里应该有一张图" — **这不是生图**

**生图流程（构建时强制执行）：**

1. **为每种人格编写详细的生图提示词**：
   - 以画风基底提示词为前缀
   - 追加该人格的具体角色描述（服装颜色/配饰/手持物/表情/姿态/背景元素）
   - 提示词必须足够详细，**至少 30 词**
2. **调用生图模型生成图片**：
   - 必须实际调用生图工具/API，获得图像数据
   - 如果当前环境有可用的生图工具，必须使用
   - 如果没有可用的生图工具，必须**停下来告知用户**，请求用户提供生图能力，**不可以用 CSS 画一个替代品然后继续**
3. **将图片内嵌到 HTML**：
   - 以 base64 编码嵌入 `<img>` 标签的 src 中
   - 或以 data URL 形式嵌入 CSS background-image 中
   - **必须是真实的图片数据，打开 HTML 可直接看到图片**

**生图数量要求**：
- 每种人格 1 张形象图，全部必须是 AI 生成的真实图片
- 总计：人格数量 张（10-16 张）

**如果无法生图怎么办**：
- **不可以**静默地用 CSS 画一个"替代品"然后当作完成了
- **必须**停下来，明确告诉用户："我需要生图能力来生成人格形象图，但当前没有可用的生图工具。请提供生图工具/API，或者告诉我如何调用。"
- 在生图能力可用之前，可以先完成其他所有部分，但形象图位置必须留空占位，标注"待生图"
- **绝对不可以用 CSS/SVG/Canvas 画一个图形冒充生图结果**

## 八、页面设计

### 页面流程

```
首页（标题 + 开始按钮 + 粒子效果）
    ↓
答题页（35 道题，带进度条）
    ↓
结果页（人格名 + 形象图 + 维度条 + 深度解读 + 兼容性 + 分享）
```

### 首页

- 页面底色由风格套餐决定
- 游戏标题（主题化字体设计，大字，有字体特效）
- 副标题/引导语（如"发现你在 XX 世界中的人格"，字号明显小于主标题，轻量字重）
- "开始测试"按钮
- 粒子效果

**首页字体层级**：
- 主标题字号 ≥ 36px（移动端）/ ≥ 48px（桌面端），有 text-shadow 或渐变填色等视觉特效
- 副标题字号约为主标题的 40-50%，字重轻，颜色比主标题浅，体现层次
- 按钮文字字号居中，字间距略宽（letter-spacing），有厚重感

**首页进场动画**（页面初次加载时依次触发）：
1. 主标题从下方上浮淡入（translateY 20px → 0，500ms，delay 100ms）
2. 副标题淡入（opacity 0 → 1，400ms，delay 400ms）
3. 按钮从下方上浮淡入（translateY 10px → 0，400ms，delay 600ms）
- 所有动画 ease-out，总时长控制在 1.2s 以内

**按钮动效**：
- 常态：有轻微脉动/呼吸效果（scale 1 → 1.03 → 1，循环，3s 周期），引导用户点击
- hover：背景色加深或亮度提升，轻微放大（scale 1.05），有 transition
- 点击（active）：短暂压缩（scale 0.97），模拟按压感

**标题持续动效（进场完成后循环，选一种）**：

> 根据书的气质从下表选择一种，实现为循环播放的 CSS animation 或 JS 动画，周期不低于 3s，不能过于抢眼或分散视线。

| 效果 | 描述 | 适合气质 |
|------|------|---------|
| 渐变色流动 | 标题文字的颜色在主题色系内缓慢流动，像光在文字上扫过 | 梦幻、魔法、现代 |
| 呼吸发光 | 标题整体或文字描边有节奏地明暗脉动（周期 3-4s） | 神秘、沉静、科幻 |
| 逐字微浮 | 每个字符以轻微的相位差上下微浮，形成波浪感 | 文学、轻盈、诗意 |
| 光扫效果 | 一道高亮光线周期性地从左到右扫过标题（周期 4-6s） | 科技、金属质感、大气 |
| 打字机循环 | 副标题文字逐字打出，打完停留片刻后消隐，再循环 | 悬疑、叙事感、文字游戏 |
| 颜色渐变循环 | 标题文字在 2-3 种主题色之间缓慢过渡（非闪烁，是渐变） | 活泼、青春、流行 |
| 粒子围绕 | 数个小粒子/光点在标题周围缓慢游荡，不遮挡文字 | 奇幻、宇宙、灵动 |
| 文字投影抖动 | 标题的投影（text-shadow）有极轻微的随机位移，像烛光摇曳 | 古典、神秘、哥特 |

**首页 → 答题页过渡**：
- 点击"开始测试"后，整个首页内容淡出（300ms），答题页内容淡入，不瞬切
- 过渡期间底色保持不变

### 答题页

- **进度条**：顶部显示当前进度（如 12/35），有平滑动画
- **题目区域**：居中展示题目文字
- **选项区域**：2 个选项按钮，上下或左右排列
- 选项点击后自动进入下一题（有过渡动画）
- 不可回退修改——选了就继续
- 粒子效果持续
- 底色与首页一致，不变化

### 结果页（核心，必须精美）

> 结果页是整个游戏的高潮，玩家会截图分享。必须精美到社交媒体级别。

- 底色与首页、答题页一致

**结果页内容（按顺序依次展示）**：

1. **人格代号 + 人格名称**：大字居中，主题化字体
2. **人格形象图**：AI 生成的人物形象，居中展示
3. **四维度倾向条**：4 个进度条，显示每个维度的百分比倾向
4. **核心描述**：80-150 字的人格描述
5. **优势与盲区**：分两列展示
6. **最接近的角色**：书中哪个角色/概念 + 为什么
7. **兼容性**：展示 2-3 种关系类型（由开发者根据主题设计），每种关系对应一个人格 + 原因
8. **分享按钮**

### 分享功能

- 点击分享按钮，生成一张固定尺寸的分享卡片（适合截图，如 375×667 或 1080×1920）
- 分享卡片内容：人格代号 + 人格名 + 形象图 + 四维度倾向条 + 核心描述（简短版） + 游戏名称
- 提供提示文字引导用户截图保存

## 九、整体 UI 设计要求

> 所有界面禁止使用浏览器默认样式。UI 的视觉风格由所选 UI 风格套餐统一决定。

- 首页、答题页、结果页的视觉风格必须统一——同一套风格贯穿全程
- 卡片、按钮、进度条的造型和颜色严格遵循所选风格套餐的定义
- 标题必须有主题化字体效果（CSS text-shadow / background-clip: text / stroke 等）
- 结果页是最重要的页面，设计精度要最高
- 禁止使用默认 `<button>` 样式和浏览器默认色板

### 答题页通用约束

- 题目文字字号 ≥ 18px，行高宽松（1.6-1.8）
- 选项按钮高度足够触摸（≥ 54px）
- 选项文字与题目文字有明显字号或字重差异
- 进度条上方或旁边显示题号（如 12 / 35）
- 当前维度可视化提示（小图标或维度名标签），让玩家知道这道题属于哪个维度

### 结果页通用约束

> 结果页是整个游戏的高潮，必须精美到社交媒体级别，玩家会截图分享。

- 结果页不使用普通流式列表——内容分区必须有视觉层次感
- 人格名/代号区域单独占一个视觉焦点区，大字、主题色、有字体特效
- 各内容模块（形象图、维度条、描述、兼容性）之间用分隔线或留白明确区分，不粘连
- 形象图居中，四周有足够留白，图片容器有主题化造型（圆形框、多边形裁切、或带装饰边框）
- 4 个维度条必须逐条渐显（有 stagger 延迟），每个维度用自己的代表色，两端标注两极名称，中间显示百分比
- 核心描述文字区有轻微背景（卡片或色块），不直接浮在底色上
- 优势/盲区分两列，用图标（✦ ✗ 或主题化符号）前置
- 兼容性用卡片网格展示，每张卡片有关系类型标签 + 对方人格名 + 一句原因
- 结果页整体色彩密度高于答题页——维度色饱和度更高、对比更明显

### 内容验收

- [ ] 4 个维度从书的内容中提炼，不是照搬 MBTI
- [ ] 维度命名贴合书的气质
- [ ] 35 道题均匀覆盖 4 个维度（每维度 8-9 题）
