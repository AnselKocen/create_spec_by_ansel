# 多结局选择游戏 Spec

## 一、产品概览

| 维度 | 值 |
|------|-----|
| 产品名称 | {{GAME_NAME}} |
| 来源素材 | {{BOOK_NAME}} |
| 输出模式 | fixed-content |
| 产品类型 | 分支叙事选择游戏 / 多结局文字冒险 |
| 核心主题 | 由随附内容文件提取后确定，不在本 spec 中预设固定题材 |
| 目标体验 | 玩家通过选择、失败、回看线索和重试，逐步理解内容规则并抵达好结局 |
| 交互方式 | 阅读推进、检查线索、做出选择、触发结局、查看结局档案、从关键节点重试 |
| 内容规模 | 建议 4-7 个章节 / 阶段；至少 5 个结局，推荐 5-8 个结局 |
| 结局结构 | 多个坏结局 + 1 个好结局 + 可选隐藏结局；坏结局必须反哺后续推理 |
| 生图策略 | 封面图 + 结局图：固定生成 1 张真实 AI 封面图，并为每个结局生成 1 张真实 AI 结局图；不生成章节 CG、普通场景插图或 UI 贴图 |
| 准确性模式 | 按内容类型决定：科学、历史、真实人物、医学、法律使用 strict；科普、历史改编、人物体验使用 semi-strict；虚构、恐怖、奇幻、人格测试使用 fantasy |
| 运行形态 | 最终用户直接游玩一个已填充完成的产品；不得把内容读取、主题生成、生图或填充流程暴露为前端功能 |

本 spec 定义的是“可被不同 `skills/` 内容填充的多结局选择游戏结构”。它不是某一个固定题材游戏，不预设世界观、角色、地点、剧情或结局名称。构建阶段必须从随附内容文件中提取具体主题和内容，再把这些内容设计成分支、线索、变量、结局和视觉系统。

### 必须提取的内容

构建阶段必须从每个内容文件全文中提取以下信息：

| 提取项 | 用途 |
|--------|------|
| 核心主题 / 核心对象 | 决定游戏标题、章节主题、视觉方向和结局主题 |
| 关键人物 / 概念 / 条目 / 知识点 | 转化为角色、事件、线索、选择题境或场景对象 |
| 因果关系 / 规则 / 约束 | 转化为隐藏变量、分支判断和好结局条件 |
| 常见误区 / 禁忌 / 反例 / 失败案例 | 转化为坏结局触发条件和失败线索 |
| 关键物品 / 场景 / 符号 / 视觉意象 | 转化为 UI 主题、SVG 装饰、粒子类型和可能的生图资产 |
| 语气 / 文风 / 时代背景 | 决定叙事文本、按钮文案、结局档案和整体氛围 |

### 任意内容的提取与玩法映射方法

不同随附内容不应按固定品类套模板。实现者必须先理解内容自身的结构，再决定它适合被设计成怎样的分支、选择和结局。

| 提取维度 | 需要回答的问题 | 转化为游戏设计 |
|----------|----------------|----------------|
| 核心对象 | 这份内容真正围绕什么人、事、物、概念、系统或关系展开？ | 玩家目标、章节主轴、关键角色 / 对象 / 概念 |
| 目标与欲望 | 内容中的主体想达成什么？什么算成功，什么算失败？ | 主线目标、好结局条件、阶段性任务 |
| 规则与约束 | 内容里有哪些不能违反的规则、条件、边界、顺序或因果？ | 隐藏变量、分支门槛、失败触发条件 |
| 冲突与取舍 | 内容中有哪些两难、误判、代价、矛盾或价值冲突？ | 2-3 选一的关键选择节点 |
| 信息差 | 哪些信息一开始不能全告诉玩家，但可以通过失败或探索逐步揭示？ | 线索系统、档案解锁、坏结局反哺机制 |
| 常见错误 | 读者、学习者或角色最容易在哪里误解、走偏或做错？ | 坏结局、错误路线、失败线索 |
| 关键转折 | 哪些内容会改变玩家对目标、规则或人物的理解？ | 中段反转、章节转折、隐藏路线入口 |
| 主题回收 | 内容最终希望玩家理解什么、记住什么或重新看见什么？ | 好结局叙事、隐藏结局意义、结局档案总结 |
| 视觉符号 | 哪些物件、场景、材料、色彩、动作或符号最能代表内容？ | UI 风格、SVG 装饰、粒子、可能的生图资产 |
| 语气与节奏 | 内容适合紧张、温柔、理性、荒诞、庄重还是轻快？ | 文案语气、转场速度、按钮反馈、整体氛围 |

映射时必须遵循以下顺序：

1. 先提取内容中的核心对象和核心目标，不急着写剧情。
2. 再找出规则、约束、误区和取舍，把它们转化为隐藏变量与选择后果。
3. 用坏结局呈现错误理解或错误路线，并让每个坏结局解锁一条可复用线索。
4. 用好结局回收内容的核心规则、主题或价值，而不是只给玩家一个“胜利”画面。
5. 如设置隐藏结局，必须来自内容中的深层主题、反例、边界条件或特殊组合。
6. 章节、选项和结局名称都应从内容本身提炼，不得预设固定题材或复用通用剧情壳。

### 禁止事项

- 不得只依赖索引摘要。
- 不得只读文件名或标题。
- 不得根据标题自行编造核心内容。
- 不得把一个固定题材强行套到所有内容文件上。
- 不得把读取内容文件、提取主题、生成章节或生成图片做成最终用户可见的前端功能。

## 三、目标用户

| 用户维度 | 描述 |
|----------|------|
| 核心用户 | 喜欢选择叙事、多结局探索、失败复盘和隐藏路线的玩家 |
| 使用场景 | 单次 10-25 分钟体验；可通过多轮重试解锁全部结局 |
| 设备偏好 | 手机浏览器、桌面浏览器和平板均可体验；触屏和鼠标点击都必须顺畅 |
| 知识门槛 | 由内容文件决定；若为学习或真实内容，首轮体验应让新手也能理解基本目标 |
| 重玩动机 | 通过坏结局线索、结局档案、隐藏结局和未解锁比例促使玩家重试 |
| 体验边界 | 不要求最终用户知道 `skills/`、内容填充、生图工具或后端构建流程 |

## 四、产品目的

本产品的核心价值是：把随附内容文件中的主题、规则、误区、人物或知识点转化为一个可探索的多结局选择游戏，让玩家通过失败与重试理解内容中的关键因果，并最终主动推理出好结局。

### 成功标准

| 标准 | 可验收条件 |
|------|------------|
| 分支成立 | 至少 5 个结局可被不同选择路径触发 |
| 失败有价值 | 每个坏结局都解锁一条可用于后续推理的失败线索 |
| 好结局可推理 | 好结局不能靠随机选择达成，必须依赖前文线索和内容规则 |
| 内容可追溯 | 章节、选择、变量、结局和视觉意象均能追溯到完整内容文件 |
| 重玩动机明确 | 结局档案显示已解锁状态、路线回顾和未完成探索空间 |
| 玩法边界清晰 | 最终用户打开后直接玩完成品，不出现上传、生成、换画风、重生成等工具流程 |

## 五、视觉风格规范

视觉风格必须由产品主题和内容文件共同决定，不得使用固定通用皮肤。实现者必须先分析内容文件，再确定整体气质、配色、排版、UI 造型、粒子和可能的图像风格。

### 风格确定流程

1. 读取全部内容文件，提取主题、时代、场景、人物气质、关键物品、情绪强度和视觉符号。
2. 选择 3-5 个视觉关键词，例如“档案、潮湿纸张、蓝色实验光、童话森林、古籍边框、图鉴标签”。
3. 制定一套主色、辅助色、强调色和危险 / 成功色。
4. 设计标题、按钮、选择卡、结局卡、档案页、线索碎片和章节过渡样式。
5. 根据内容决定粒子 / 氛围层，数量控制在 10-30 个，不遮挡核心文字。

### 画风候选库与组合方法

画风不能只从少数内容类型里选择。实现者应先从内容文件中提取主题、情绪、媒介感、时代感、材质和关键符号，再从下表选择 1 个主画风和 1-2 个辅助质感组合成统一视觉方向。

| 画风方向 | 适合表达 | 可提取的视觉关键词 |
|----------|----------|--------------------|
| 暗色写实插画 | 压迫、悬疑、沉重选择、现实危机 | 低饱和、电影光、深阴影、真实材质 |
| 水墨 / 墨迹 | 东方气质、含蓄叙事、规则与留白 | 宣纸、墨痕、飞白、印章、山水轮廓 |
| 复古版画 | 命运感、古籍、传说、历史档案 | 木刻线、网纹、旧纸、双色印刷 |
| 老照片 / 胶片 | 记忆、追溯、传记、失真真相 | 颗粒、划痕、褪色、暗角、曝光痕 |
| 档案拼贴 | 调查、推理、证据链、多线索回看 | 便签、红线、文件夹、印章、索引卡 |
| 极简线条 | 抽象主题、哲学选择、轻量叙事 | 单线、留白、细边框、符号化人物 |
| 几何拼贴 | 人格、策略、抽象系统、现代议题 | 色块、圆形、三角、网格、构成主义 |
| 扁平矢量 | 清晰知识、工具感、轻松选择 | 明快色块、干净图标、信息图层级 |
| 像素艺术 | 复古游戏、循环挑战、轻量冒险 | 低分辨率、有限色板、方块边缘 |
| 赛博 / 故障 | 技术、未来、监控、身份错位 | 霓虹、扫描线、数据噪声、RGB 偏移 |
| 科学图谱 | 实验、系统、规则、观察与验证 | 坐标网格、标注线、公式、仪表、剖面 |
| 自然手账 | 探索、收集、成长、观察自然 | 手写注释、纸张纹理、标本框、水彩 |
| 童话绘本 | 温柔叙事、寓言、成长、奇遇 | 手绘边缘、柔和色块、纸页、微光 |
| 新艺术海报 | 花卉、神秘、装饰性强的主题 | 曲线纹样、植物边框、优雅构图 |
| 波普 / 孟菲斯 | 轻快、荒诞、反差、幽默选择 | 高饱和、波点、锯齿、夸张几何 |
| 黑白高对比 | 严肃抉择、道德冲突、剪影化叙事 | 大块黑白、硬边阴影、负形空间 |
| 低多边形 / 3D 玩具 | 轻策略、可爱冒险、轻松实验 | 圆润形体、软光、简化材质 |
| 纸艺 / 剪纸 | 民俗、手作、舞台感、寓言化表达 | 纸层、阴影、剪影、折痕、镂空 |
| 金属 / 工业 | 机械、秩序、压力系统、硬核规则 | 铆钉、刻度、钢铁、警示色、磨损 |
| 玻璃拟态 | 梦境、界面感、流动记忆、轻科技 | 半透明、模糊背景、细描边、光斑 |

选定画风后必须写出：

- 主画风名称。
- 3-5 个视觉关键词。
- 主色、辅助色、强调色、危险色、成功色。
- 标题、按钮、选项卡、结局卡、档案页分别如何体现该画风。
- 粒子 / 氛围层使用哪一种主题化运动或静态纹理。
- 如果需要生图，`STYLE_PROMPT_BASE` 必须与主画风一致。

### UI 美化要求

- 游戏标题不能只是普通 `<h1>`，必须使用主题化字体效果，例如 `text-shadow`、`background-clip: text`、描边、轻微发光、印刷纹理或 SVG 装饰。
- 按钮必须像内容主题中的物件或符号，例如票根、实验标签、档案贴纸、图鉴书签、信件封条或人格徽章；禁止浏览器默认按钮样式。
- 选择按钮必须有 hover、active、selected、disabled 状态，且反馈不能只靠颜色变化。
- 结局页必须有独立视觉层级，坏结局、好结局、隐藏结局在色彩、图标或边框上可区分。
- 档案页必须像一个可回看的系统，而不是普通列表。
- 页面切换不能瞬切，必须有 200-800ms 的淡入、滑入、翻页、墨迹扩散、扫描线或主题化过渡。

### SVG / CSS 技巧

- 使用内联 SVG 做主题化图标、章节符号、结局徽章、线索碎片、进度环或装饰边框。
- 使用 `currentColor` 让 SVG 图标继承主题色。
- 使用 CSS `clip-path`、渐变遮罩、边框纹理、`box-shadow`、`filter`、`backdrop-filter` 等方式增强 UI 质感。
- 使用 SVG `stroke-dasharray` 或 CSS 变量实现结局收集进度。
- 未解锁结局必须有主题化遮罩，例如锁印、模糊档案、未知标本、残缺页、未显影照片或剪影，而不是普通灰框。

### 粒子 / 氛围设计

粒子必须服务内容主题：

| 粒子类型 | 适用方向 |
|----------|----------|
| 尘埃 / 纸屑 | 档案、历史、旧书、悬疑 |
| 星点 / 光尘 | 童话、幻想、宇宙、人格抽象 |
| 气泡 / 微粒 | 化学、海洋、生物、饮品 |
| 雨线 / 雾气 | 恐怖、悬疑、忧郁叙事 |
| 几何碎片 | 科技、人格、抽象选择 |
| 花瓣 / 落叶 | 自然、传记、温柔叙事 |

约束：

- 粒子数量通常为 10-30 个。
- 粒子层必须 `pointer-events: none`。
- 粒子不得遮挡正文、选项和结局信息。
- 必须支持 `prefers-reduced-motion`，在用户减少动态效果时禁用漂移动画或替换为静态纹理。

## 六、封面图与结局图生图策略与资产契约

本产品固定需要真实生图资产，但生图范围必须严格收窄：**固定生成 1 张封面图 + 每个结局 1 张结局图**。除封面标题页与结局页外，不生成章节 CG、普通场景插图、线索图、档案图、按钮图或 UI 贴图。

必须生成：
- 封面图 1 张，用于封面标题页。
- 每个结局 1 张结局图，用于对应结局页和解锁后的结局档案。
- 如果最终游戏有 5 个结局，则生成 1 张封面图 + 5 张结局图。
- 如果最终游戏有 8 个结局，则生成 1 张封面图 + 8 张结局图。

普通游戏过程的视觉由主题化 UI、SVG/CSS、粒子、档案系统、结局卡和文字排版承担。真实位图只用于封面标题页和结局相关页面。

#### 什么是生图

- 调用生图模型或图像生成 API / 工具，输入提示词，获得一张真实的图像文件。
- 图片数据必须在构建阶段获得，并能在最终 HTML 中直接显示。

#### 什么不是生图

- 用 CSS 渐变、`box-shadow`、`border-radius` 画出来的图形不是生图。
- 用 SVG path、circle、polygon 手动描绘的图形不是生图。
- 用 Canvas 代码绘制的图形不是生图。
- 用 emoji 或 Unicode 符号拼凑不是生图。
- 用纯文字描述“这里应该有一张图”不是生图。
- 用 CSS / SVG / Canvas 做一个风格化占位图然后当作图片不是生图。

#### 生图流程

生图是构建闸门，不是视觉建议。如果本项目声明需要生图，开发者必须先完成真实图片生成，再进入依赖该图片的页面实现与最终交付。不能把“待生图”“以后补图”“用 CSS 临时代替”当作完成状态。

1. 根据内容主题确定统一 `STYLE_PROMPT_BASE`。
2. 为封面图编写完整提示词，完整提示词 = `STYLE_PROMPT_BASE` + 游戏核心主题、关键视觉符号、封面构图、情绪基调、光影和留白区域。
3. 为每个结局编写完整提示词，完整提示词 = `STYLE_PROMPT_BASE` + 该结局的最终状态、核心情绪、关键角色 / 对象、象征物、光影和构图。
4. 每张图的提示词至少 30 个英文词或等量详细描述。
5. 实际调用可用的生图工具或 API。
6. 将生成结果以本地静态资源、base64 或 data URL 形式绑定到数据中。
7. 如果最终交付物是单个 HTML 文件，必须先生成真实图片，再转换为 base64 / data URL 内嵌。

#### 封面图设计要求

- 封面图必须建立游戏主题、核心情绪和主要视觉符号，但不得剧透具体结局条件或隐藏路线。
- 封面图不得包含可读文字、logo、水印或伪 UI；标题、按钮和入口信息必须由 HTML/CSS 叠加实现。
- 封面图构图必须为标题和开始按钮预留安全区域，移动端裁切后仍能保留主体。
- 封面图需要能配合进入动效使用，例如轻微推近、视差、柔焦淡出或主题化遮罩过渡。

#### 结局图设计要求

- 坏结局图应表达错误理解、错误路线或核心误区造成的最终状态，但不要依赖血腥、低俗或纯惊吓。
- 普通 / 中性结局图应表达部分达成与代价并存。
- 好结局图应表达目标达成和主题回收。
- 隐藏结局图应表达深层主题、反常路线或特殊组合的独立意义。
- 结局图不得提前出现在普通章节中，只能在对应结局页和已解锁后的结局档案中展示。

### IMAGE_ASSET_MANIFEST

必须在实现核心页面前输出并核对 `IMAGE_ASSET_MANIFEST`。封面图 1 条资产记录，每个结局 1 条图片资产记录，每张图片至少包含：

| 字段 | 说明 |
|------|------|
| `id` | 图片唯一 ID，如 `cover_image`、`ending_bad_01`、`ending_good`、`ending_hidden` |
| `purpose` | 图片用途，固定为 `cover_image` 或 `ending_cg` |
| `ending_id` | 仅当 `purpose = "ending_cg"` 时必填，对应的结局 ID，必须能关联到 `endings[].id`；封面图可省略或为 `null` |
| `style_prompt_base` | 本项目统一画风基底 |
| `prompt` | 该图片的完整正向提示词 |
| `negative_prompt` | 明确排除项，如 text、logo、watermark、extra fingers、foreground clutter |
| `aspect_ratio` | 目标比例，如 16:9、4:3、1:1 |
| `output_path_or_data_url` | 真实生成结果的文件路径或 data URL |
| `status` | 必须为 `generated` 才算通过 |

生成闸门必须满足：

- 所有必需图片的 `status` 都是 `generated`。
- 必须有且只有 1 张 `purpose = "cover_image"` 的封面图。
- 每个结局都有且只有 1 张对应结局图。
- 不存在章节 CG、普通场景插图、线索图、档案图、按钮图、UI 贴图等额外图片资产。
- 每个 `output_path_or_data_url` 指向真实位图文件或真实位图数据。
- 最终页面引用的是这些真实图片，而不是 CSS / SVG / Canvas / emoji / 文字占位。
- 图片生成失败时必须重写提示词并重试；连续失败后停下来报告失败资产 ID、失败原因和已尝试提示词。

如果没有可用生图能力，而本产品又需要真实图片，必须明确告诉用户：“我需要生图能力来生成这些图片，但当前没有可用的生图工具。请提供生图工具/API，或者告诉我如何调用。”在生图能力可用之前，不得交付声称完成的最终成品。

## 八、用户动线

### 核心路径

```text
封面标题页（AI 封面图 + 开始 / 继续 / 档案）
→ 点击开始 / 继续
→ 进入动效
→ 内容驱动的开场章节 / checkpoint
→ 阅读场景
→ 检查线索
→ 做出选择
→ 路线推进
→ 坏结局 / 普通结局 / 好结局 / 隐藏结局
→ 结局档案更新
→ 从关键节点重试或重新开始
→ 逐步解锁更深路线
```

### 首次游玩路径

1. 玩家打开封面标题页，看到真实 AI 生成封面图、主题化标题和开始入口。
2. 玩家点击开始，系统播放进入动效后进入已完成的游戏。
3. 开场章节建立主题、玩家身份、目标和第一个冲突。
4. 玩家阅读场景，检查 0-3 个线索。
5. 玩家做出选择，系统展示短期反馈。
6. 玩家进入后续章节或触发第一个坏结局。
7. 第一次坏结局后，系统解锁结局档案入口，并提示本次失败留下了可回看的线索。

### 重玩路径

1. 玩家从封面标题页或结局页进入结局档案。
2. 玩家查看已解锁结局、关键选择路径和失败线索。
3. 玩家从最近关键节点或章节起点重试。
4. 已读文本可加速，但新文本必须正常展示。
5. 玩家利用已知线索避开坏结局，逐步逼近好结局或隐藏结局。

### 完成路径

1. 玩家理解核心规则或关键内容点。
2. 玩家在终局前满足好结局变量或路线条件。
3. 系统展示好结局，回收主要主题和关键内容。
4. 若存在隐藏结局，结局档案给出不剧透的未完成提示，例如“仍有一个异常记录未归档”。

## 十一、交互设计

### 操作映射

| 用户操作 | 状态变化 | 反馈方式 |
|----------|----------|----------|
| 点击开始 | 从封面标题页进入开场章节 | 封面图轻微推近 / 视差 / 淡出，标题和入口退场，章节标题出现 |
| 点击继续 | 当前文本推进到下一段或下一场景 | 文本渐显、滚动定位、按钮 active 反馈 |
| 点击线索 | 展开线索详情并记录已查看 | 线索高亮、图标变化、轻微粒子或边框动画 |
| hover / 聚焦选项 | 选项进入可选中状态 | 边框、阴影、位移、图标或背景纹理变化 |
| 点击选项 | 更新变量、路线标签和选择历史 | 短暂停顿、选择反馈、进入下一场景 |
| 达成坏结局 | 解锁结局和失败线索 | 结局页出现、档案新增提示 |
| 查看档案 | 从当前页面打开档案 | 面板滑入或翻页效果 |
| 回到关键节点 | 恢复 checkpoint 场景和当时状态 | 过渡动画、已读文本可加速 |
| 切换设置 | 修改阅读或动效偏好 | 即时应用、toast 提示 |

### 状态反馈要求

- entry transition：点击开始 / 继续后必须播放 600-1200ms 的主题化进入动效，不能瞬间切换白屏或直接跳到正文；减少动态效果时改为 200ms 内的透明度过渡。
- hover：选项、按钮、线索、档案卡必须有明确 hover 或 focus-visible 样式。
- active：点击时必须有压下、缩放、亮度、边框或短动画反馈。
- selected：已选择选项要短暂保留选中状态，再进入反馈页。
- disabled：不可选路线必须说明原因或以主题化方式表现为未解锁。
- success：解锁好结局、隐藏结局、关键线索时必须有正向反馈。
- error / bad：触发坏结局时不要只显示失败文字，必须展示失败线索和重试入口。
- archive update：结局档案更新时必须有 toast、徽章闪烁或档案页新增动效。

### 阅读体验

- 正文字号移动端不低于 16px，桌面端建议 17-20px。
- 正文行高不低于 1.6。
- 每段文本宽度控制在适合阅读的范围，桌面端主文本区不超过 760px。
- 首次阅读不能默认跳过；已读内容可以提供加速。
- 所有交互必须支持键盘 focus 和 Enter / Space 激活。

## 十二、核心机制约束

### 不可更改的玩法规则

1. 本产品必须让最终用户直接体验已完成的多结局游戏。
2. 具体题材、角色、章节、结局名称和选项文案必须由随附内容文件驱动，不得预设固定主题。
3. 最终游戏至少包含 5 个结局，其中必须包含多个坏结局和 1 个好结局。
4. 每个坏结局都必须解锁一条可复用失败线索。
5. 好结局必须依赖玩家对前文线索、内容规则或关键因果的理解，不能靠随机选择碰出。
6. 如果设置隐藏结局，隐藏结局必须有独立意义，不能只是好结局换文案。
7. 结局档案必须存在，并记录结局、路线回顾和失败线索。
8. 选择不能写成明显的“正确 / 错误”，必须体现真实取舍。

### 内容文件提取与映射约束

构建阶段必须完成：

- 读取每个内容文件全文。
- 提取核心主题、关键对象、规则、误区、视觉意象和语气。
- 将提取出的内容映射为章节、选择、隐藏变量、结局触发条件和档案文本。
- 保留 `sourceRefs`，让关键内容可以追溯到文件路径和提取依据。
- 对于科学、历史、传记、医学、法律等内容，必须校验事实、公式、时间线、人物关系和术语。

### 准确性模式约束

| 模式 | 适用内容 | 要求 |
|------|----------|------|
| strict | 科学、历史、医学、法律、真实传记、教材类内容 | 不允许虚构关键知识；公式、单位、时间线、人物关系和答案必须正确 |
| semi-strict | 科普、历史改编、人物体验、轻学习游戏 | 核心事实和概念必须正确，允许叙事化表达但不能改变关键因果 |
| fantasy | 恐怖、奇幻、人格测试、抽象游戏、非真实世界设定 | 可原创设定，但必须与内容文件有内在联系并保持内部一致 |

### 玩法边界约束

读取内容文件、提取主题、设计章节、生成图片、填充数据、准确性检查和自检都属于构建阶段动作。最终用户界面中不得出现以下内容：

- 上传内容文件。
- 粘贴文本生成游戏。
- 选择模型或 API。
- 点击生成章节。
- 重新生成结局。
- 换画风生成。
- 查看后台提取流程。

最终用户只能看到完成后的游戏体验：开始、阅读、选择、检查线索、触发结局、查看档案、重试、存档和设置。

## 十三、内容 / 章节概览

最终产品建议使用 4-7 个章节 / 阶段。具体章节标题和内容由 `skills/` 文件提取生成。

| 阶段 | 章节职责 | 内容来源 | 设计要求 |
|------|----------|----------|----------|
| 开场 | 建立主题、玩家身份、目标和第一个冲突 | 内容文件中的核心主题、主要对象、开端信息 | 不暴露所有规则，只给玩家一个明确目标 |
| 学习段 1 | 引入第一类选择后果 | 内容文件中的基础规则、初级误区、主要人物或对象 | 至少可导向 1 个早期坏结局 |
| 学习段 2 | 引入更复杂的规则或动机 | 内容文件中的因果、冲突、实验条件、人物关系或条目关系 | 坏结局开始提供可复用线索 |
| 转折段 | 揭示核心误区或反转信息 | 内容文件中的关键反例、禁忌、转折、边界条件 | 玩家开始理解好结局所需条件 |
| 收束段 | 综合考验前面学到的信息 | 前面章节已出现的规则、线索和选择后果 | 多个变量共同影响结局 |
| 终局段 | 进入坏结局、好结局或隐藏结局 | 内容文件中的核心价值、结论、寓意或知识收束 | 必须回收主要线索与主题 |

### 结局规模

| 类型 | 数量 | 要求 |
|------|------|------|
| 坏结局 | 至少 3 个，推荐 4-6 个 | 每个坏结局揭示一个可复用线索 |
| 普通 / 中性结局 | 可选 0-2 个 | 玩家达成部分目标，但留下代价或缺憾 |
| 好结局 | 必须 1 个 | 玩家理解核心规则并完成主要目标 |
| 隐藏结局 | 推荐 1 个 | 需要额外探索、特殊路线或深层理解 |

### 结局内容要求

每个结局必须包含：

- 结局标题。
- 结局类型。
- 触发条件。
- 结局叙事。
- 关键选择回顾。
- 来源内容引用。
- 解锁线索或档案记录。
- 重试或查看档案入口。

## 十四、技术约束

| 维度 | 要求 |
|------|------|
| 技术栈 | 单页 HTML / CSS / JavaScript，或等效前端实现；不得依赖服务器才能完成核心体验 |
| 部署方式 | 可作为静态页面运行 |
| 存储 | 使用 localStorage 保存进度、结局档案、设置和已读状态 |
| 外部依赖 | 核心体验不得依赖网络；若生图已在构建阶段完成，最终页面只引用内嵌或本地资产 |
| 响应式 | 必须适配手机、平板和桌面 |
| 性能 | 初次加载后操作反馈应在 100ms 内出现；动画目标 30fps 以上 |
| 可访问性 | 支持键盘操作、focus-visible、足够对比度、可调整字号、减少动态效果 |
| 数据安全 | 不在前端暴露任何 API key、模型密钥或构建阶段工具配置 |
| 可维护性 | 游戏数据、章节、选项、结局、变量和资产清单应结构化，不写散乱硬编码 |
| 离线能力 | 完成构建后的产品应能在无网络环境下体验完整核心流程 |

### 响应式布局要求

- 移动端使用单列结构：正文、线索、选项纵向排列。
- 桌面端可使用主叙事区 + 侧边档案 / 状态栏，但不得让阅读区过宽。
- 选项按钮在触屏设备上高度不低于 44px。
- 结局档案在小屏幕上应使用可折叠卡片，不得横向溢出。
- 所有文本必须在容器内换行，不得遮挡按钮或后续内容。

## 十五、开发指引

### 执行步骤

1. **读取完整内容文件**  
   读取 `skills/` 中每个内容文件全文，建立 `ContentSource` 记录。不得只看摘要、标题或文件名。

2. **判断内容类型和准确性模式**  
   根据内容类型选择 `strict`、`semi-strict` 或 `fantasy`。科学、历史、医学、法律、真实传记必须使用 `strict`。

3. **提取内容结构**  
   提取主题、关键对象、规则、误区、视觉意象、语气和可转化为选择后果的因果链。

4. **设计章节和核心循环**  
   生成 4-7 个章节 / 阶段。每章必须服务多结局选择玩法，不得写成纯阅读电子书。

5. **设计选择和变量**  
   建立 3-5 个隐藏变量。每个关键选择都必须有变量影响、路线标签或结局条件影响。

6. **设计结局**  
   至少设计 5 个结局。每个坏结局必须解锁失败线索，好结局必须可由线索推理得出。

7. **设计结局档案**  
   档案必须记录已解锁结局、失败线索、路线回顾和未解锁状态。不得直接给完整攻略。

8. **执行封面图与结局图生图策略**  
   先生成 1 张真实封面图，再为每个结局生成 1 张真实结局图，完成 `IMAGE_ASSET_MANIFEST` 后再实现封面标题页、结局页和结局档案；不得生成章节 CG、普通场景插图、线索图、档案图、按钮图或 UI 贴图。

9. **实现主题化 UI**  
   根据内容文件确定视觉风格，实现封面标题页、进入动效、主题化标题、按钮、档案、线索、结局卡、SVG 装饰和粒子层。

10. **实现存档与重试**  
   使用 localStorage 保存进度、结局、线索、已读文本、设置和关键节点。

11. **执行自检**  
   按第十六章验收标准逐项检查。任何未达成项都不能标记为完成。

### 玩法开发边界

构建阶段可以读取内容、提取主题、设计分支、生成图片和填充数据，但这些流程不得出现在最终用户界面。最终用户只看到已经完成的多结局游戏。

### 文案生成要求

- 所有章节、选择和结局文案必须符合内容文件语气。
- 选择文案必须短而有取舍感，不写成答案选项。
- 坏结局文案必须给出失败原因的线索，但不直接剧透完整好结局路线。
- 好结局必须回收主要内容主题。
- 隐藏结局必须有独立意义和额外理解价值。

## 十六、验收标准

### 功能验收

- 最终产品打开后进入封面标题页，并可以直接开始游戏。
- 至少 5 个结局可以通过不同选择路径触发。
- 至少包含 3 个坏结局和 1 个好结局。
- 每个坏结局都会解锁失败线索。
- 结局档案可以查看已解锁结局、路线回顾和失败线索。
- 玩家可以从关键节点重试，不必每次完整重看所有已读内容。
- localStorage 能保存进度、结局档案、线索和设置。

### 随附内容文件验收

- 完整读取了每个内容文件全文。
- 章节、选择、变量、结局、视觉意象都能追溯到内容文件。
- 未只依赖索引摘要、文件名或标题。
- 未根据标题自行编造核心内容。
- 未把固定题材强行套到所有内容文件上。

### 生图验收

- 必须存在 `IMAGE_ASSET_MANIFEST`。
- `IMAGE_ASSET_MANIFEST` 中封面图和每个结局图资产的 `status` 必须是 `generated`。
- 必须有且只有 1 张 `purpose = "cover_image"` 的封面图。
- 每个结局必须有且只有 1 张结局图，且 `ending_id` 能对应到 `endings[].id`。
- 不存在章节 CG、普通场景插图、线索图、档案图、按钮图、UI 贴图等额外图片资产。
- 每个 `output_path_or_data_url` 必须指向真实位图文件或真实位图数据。
- 所有图片必须使用统一 `STYLE_PROMPT_BASE`。
- 图片失败时必须停止并报告，不得用占位图冒充完成。

### 视觉验收

- 封面标题页必须显示真实 AI 生成封面图，不能用 CSS / SVG / Canvas / emoji / 文字占位冒充。
- 标题不是普通 `<h1>`，有主题化字体效果或 SVG 装饰。
- 按钮、选择卡、结局卡、档案页不是浏览器默认样式。
- UI 风格与内容文件提取出的主题、时代、物件和语气一致。
- 使用了 SVG / CSS 做主题化图标、边框、进度或装饰。
- 粒子 / 氛围层存在且不遮挡文本或选项。
- 支持 `prefers-reduced-motion`。

### 交互验收

- 所有可点击元素都有 hover、focus-visible、active 状态。
- 点击开始 / 继续后必须有进入动效，封面图、标题和入口退场，开场章节入场；不能白屏或瞬间硬切。
- `prefers-reduced-motion` 下进入动效必须降级为短透明度过渡，不能保留大幅位移或视差。
- 做出选择后有短期反馈，不是直接无提示跳转。
- 坏结局页有重试和查看档案入口。
- 结局档案更新有明确反馈。
- 已读文本可以加速，新文本不能默认跳过。
- 键盘可完成主要操作。

### 数据验收

- 构建阶段数据和运行时用户进度数据分离。
- 章节、场景、选择、变量、结局和线索均结构化。
- `coverImageAssetId` 必须能对应到 `IMAGE_ASSET_MANIFEST` 中唯一的 `cover_image` 资产。
- 每个 `ending.imageAssetId` 必须能对应到 `IMAGE_ASSET_MANIFEST` 中该结局的 `ending_cg` 资产。
- 每个关键内容对象有 `sourceRefs` 或等效来源追踪。
- localStorage 数据结构稳定，刷新页面后能恢复。
- 清除存档功能不会破坏游戏原始数据。

### 响应式验收

- 手机、平板、桌面布局均可正常阅读和操作。
- 移动端按钮高度不低于 44px。
- 文本不溢出、不遮挡、不与按钮重叠。
- 档案页和结局页在小屏幕上可完整浏览。
- 横竖屏切换后核心状态不丢失。

### 准确性 / 一致性验收

- strict 模式下，事实、公式、单位、时间线、人物关系和答案必须正确。
- semi-strict 模式下，核心事实和关键因果不得被戏剧化表达扭曲。
- fantasy 模式下，原创设定必须与内容文件有内在联系并保持内部一致。
- 错误选项必须对应真实误区、合理误判或主题内因果，不能随机胡编。
- 好结局条件必须能由前文线索推理出来。

### 输出模式验收

- 产品概览中明确写有输出模式，且实现边界与该模式一致。
- 最终用户界面中不得出现上传内容、输入主题、点击生成、重新生成、换画风、模型设置、API key 等工具流程。
- 读取内容文件、提取主题、生图和填充数据只存在于构建阶段、开发指引、核心约束和验收标准中。
- 最终用户打开后直接玩、阅读、选择、解锁结局、查看档案和重试。
